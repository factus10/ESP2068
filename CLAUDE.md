# ESP2068 — TS2068 port of ESPectrum

## What this project is

A Timex Sinclair 2068 emulator, built by porting [EremusOne/ESPectrum](https://github.com/EremusOne/ESPectrum) (an ESP32 + VGA + PS/2 ZX Spectrum 48K/128K/Pentagon emulator) to run the TS2068 on the same ESP32 hardware. The 2068's SCLD chip replaces the Sinclair ULA; everything else (Z80 core, VGA DMA driver, AY sound, PS/2 keyboard, SD/file-nav, OSD menu) is reused as a chassis.

The technical design — SCLD register contract, 8-slot memory model, display modes, timing, cartridge loading, and the four-slice work breakdown — lives in [TS2068-ESPECTRUM-PORT-PLAN.md](TS2068-ESPECTRUM-PORT-PLAN.md). Treat that file as the spec every slice codes against. The phased build-order plan is in [PLAN.md](PLAN.md).

## Current repo state

ESPectrum's full source and history are vendored in (merged from the `upstream` remote, `EremusOne/ESPectrum`, `master` branch — see Phase 0 in `PLAN.md`). `src/`, `include/`, `platformio.ini` etc. now exist at repo root alongside the planning docs. To pull upstream fixes later: `git fetch upstream && git merge upstream/master`. The `nopsram` build **compiles clean with all four slices' first increments integrated** (`pio run -e nopsram`, verified 2026-07-10 — see `PLAN.md` Phase 0/2); flash-and-boot on real hardware is still unverified (no ESP32 board in this environment).

**Slice 1 (MMU + memory core) has a first real increment landed** — see `PLAN.md` Phase 2 for the full breakdown. In short: [`include/SCLD.h`](include/SCLD.h) + [`src/SCLD.cpp`](src/SCLD.cpp) now allocate and resolve real HOME ROM/RAM/DOCK/EXROM backing store (not stubs); `src/CPU.cpp` has a new `Z80Ops::is2068` flag and six `_2068` peek/poke/fetchOpcode/addressOnBus functions wired into the existing `_std`/`_2A3` function-pointer dispatch (**not** in-place edits to `_std`, which stays shared by 48K/128K/Pentagon/TK); `src/Ports.cpp` fully-decodes ports 0xF4/0xFF ahead of the Spectrum's partial-decode chain; `include/cpuESP.h` has sourced (libspectrum) 60Hz timing constants. A standalone host-side test, [`test/host/scld_test.cpp`](test/host/scld_test.cpp), exercises the real `SCLD.cpp` (30/30 checks passing) without needing ESP-IDF or hardware.

**Slice 4 (DOCK corpus + test harness) also has a first increment**: [`test/dock/`](test/dock/) has offline Python tooling for the `.DCK` cartridge container format (`dck_format.py` parser/writer, `dck_inspect.py` CLI, `make_synthetic_dck.py` fixture generator) plus a written provenance policy — no real cartridge/ROM dumps belong in this public repo; see `test/dock/README.md` before adding any binary image anywhere in the tree.

**Slice 3 (cartridge loader) has a first real increment, plus a second increment that found and fixed a real bug**: [`include/DockLoader.h`](include/DockLoader.h) + [`src/DockLoader.cpp`](src/DockLoader.cpp) parse `.DCK` files (a direct C++ port of `test/dock/dck_format.py`'s already-verified logic) and populate `SCLD`'s DOCK/EXROM chunks. Autostart deliberately does **not** use the `check_trdos` PC-watching pattern the original plan doc proposed — LROS only autostarts at boot, unlike TR-DOS, so a one-shot apply at reset is enough. Wired into `CPU::reset()`'s `"2068"` branch.

**The bug**: the first increment assumed an LROS cartridge could just be paged in and left to execute from `PC=0` (Z80::reset()'s default). Testing against a real LROS cartridge (`Zebra OS-64.BIN`, from the local reference library) proved that wrong — its real header occupies bytes 0-4 of chunk 0 as metadata, and the actual entry point is wherever the header's own start-address field points (`$0005` for Zebra, not `$0000`; landing at PC=0 would execute the header bytes as garbage instructions). Fixed: `DockLoader::autostartEntryPoint()` reads that field for real; `CPU::reset()` calls `Z80::setRegPC()` with it. [`test/host/real_cartridge_test.cpp`](test/host/real_cartridge_test.cpp) (18/18 checks) is what caught this — it wraps two **genuine** cartridge dumps (Zebra OS-64, a real LROS; `eToolkit.ROM`, a real AROS that correctly does *not* trigger autostart since it has no chunk 0) as in-memory `.DCK` buffers and runs them through the real loader, never touching the repo with the actual ROM bytes. `test/host/dockloader_test.cpp` (16/16) still covers the synthetic fixture and mechanical/error-handling cases; the real-content test is what a purely-synthetic fixture couldn't have caught, since its header bytes were never meant to resemble genuine cartridge metadata.

**Slice 2 (hi-res renderer + VGA mode table) now actually draws something, across two increments**: [`include/SCLDVideo.h`](include/SCLDVideo.h) + [`src/SCLDVideo.cpp`](src/SCLDVideo.cpp) implement and test the byte-to-pixel expansion math (standard-mode pixel-doubling, hi-res dual-display-file interleave, ink/paper decode — `test/host/scldvideo_test.cpp`, 15/15 checks). `src/ESP32Lib/VGA/VGA.h` has a new fixed 512-active-column `vidmodes[]` row (`SCLD_VGA_MODE_INDEX`), computed with the real APLL clock-divider algorithm from `tools/VideoModeTool.py`, wired into `Video.cpp`'s mode-select.

The second increment wired that math into an actual renderer, but **not** the way the first increment expected: `VIDEO::Draw`/`Draw_Opcode` are self-retargeting function pointers the existing Spectrum renderers reassign to each other as `tstates` crosses scanline boundaries (cycle-accurate rendering under memory contention) — TS2068 has no contention, so instead of building a TS2068 version of that state machine, [`VIDEO::RenderFrame2068()`](src/Video.cpp) renders the whole frame in one batch, called once per frame from `CPU::loop()`'s existing per-frame hook (`if (Z80Ops::is2068) VIDEO::RenderFrame2068(); else VIDEO::EndFrame();` — same call site every machine already used). `Draw`/`Draw_Opcode` stay pointed at the existing `&VIDEO::Blank` the whole session. See `PLAN.md`'s slice 2 writeup for what this covers (standard + hi-res modes, FLASH, BRIGHT) and doesn't (dual-file and hi-color modes render as standard-from-DFILE1, wrong but not blank; no border-color changes; no OSD compositing over a running TS2068 screen) — still entirely unverified on a real monitor, same as everything else in this repo.

**A local TS2068 reference library significantly upgraded source quality partway through this project**: `/Users/david/Documents/Projects/TS2068 Ref Library/` (local to this machine — not part of this repo, other contributors/sessions elsewhere won't have it; check it's still there before relying on a claim sourced from it). Real ROM disassemblies (HOME ROM + EXROM) and cross-referenced documentation (`docs/*.md` — memory map, video modes, dispatcher, I/O ports, errata). It already caught one real error: v0.1 of the plan doc had the hi-res video-mode bit value wrong (see `TS2068-ESPECTRUM-PORT-PLAN.md`'s "Port 0xFF" section) — though a later pass against FUSE (below) found that correction was itself incomplete; see PLAN.md's FUSE cross-reference writeup. **Contains real TS2068 ROM images** (`2068 ROMS/2068Home.BIN` + `2068Exrom.BIN` confirmed genuine — the disassembly-documented `$FF` version byte at HOME ROM offset `$0013` matches the real file — plus several alternative/homebrew ROMs like `gus-home.rom`/`gus-exrom.rom`, `eToolkit.ROM`, `Zebra OS-64.BIN`) — fine to read for reference/disassembly/testing, but per this repo's own provenance policy (`test/dock/README.md`), never copy their content into this git repo without confirming redistribution rights first. `RomLoader` (below) reads them from this path directly at test time; nothing gets copied in.

**A local FUSE 1.9.0 checkout (`/Users/david/Downloads/fuse-1.9.0`, also local to this machine, not part of this repo) is the highest-quality reference source found so far** — a mature, real, GPL, actively-TS2068-emulating codebase, not a summarized doc. `peripherals/scld.c`/`scld.h` (SCLD memory/video register emulation), `peripherals/dck.c` (`.DCK` loading), `machines/ts2068.c`/`tc2068.c`, `display.c`/`display.h` (rendering), `ui/fb/fbdisplay.c` (`uidisplay_plot16` — the exact pixel-expansion reference). A single cross-reference pass against it (2026-07-10) found **four real bugs already shipped in this repo** — see PLAN.md's "Cross-referencing FUSE" writeup for the full story: a port-address-matching bug in `Ports.cpp` that would have broken the MMU port for almost all real software, a hi-res pixel-interleave-order bug, an imprecise hi-res-submode trigger, and a standard-mode attribute-row-addressing bug that had been sitting untested inline in `Video.cpp`. All fixed and re-tested. When in doubt about SCLD/video/cartridge behavior going forward, check FUSE before trusting a summarized doc, including this project's own prior conclusions.

**ROM loading is real now**: [`include/RomLoader.h`](include/RomLoader.h) + [`src/RomLoader.cpp`](src/RomLoader.cpp) load a raw HOME ROM (16K) or EXROM (8K) binary from a file into `SCLD`'s backing store (`SCLD::loadHomeRom()` is new; EXROM reuses the existing `SCLD::loadExromImage()`). Deliberately does **not** reuse ESPectrum's existing Spectrum-machine "custom ROM" mechanism (`OSDMain.cpp`'s `updateROM()`) — that's a self-reflashing OTA-partition trick, confirmed via investigation to be too Spectrum-specific to be worth adapting; this is the much simpler "read a file, copy the bytes" shape instead. Wired into `ESPectrum.cpp`'s boot sequence for `Config::arch == "2068"`, from fixed conventional SD paths (`RomLoader::DEFAULT_HOME_ROM_PATH`/`DEFAULT_EXROM_PATH` — a bootstrapping convention, not a final OSD-integrated interface; that's still Phase 3). Host test (`test/host/romloader_test.cpp`, 13/13 checks) validates against the *real* ROM files above — confirms the documented `$FF` version-byte fact against actual loaded content, not just that a copy succeeded — and skips (doesn't fail) that part gracefully if the reference library isn't present.

**Keyboard input (port 0xFE, read side) is wired now**: `src/Ports.cpp`'s `is2068` branch of `Ports::input()` gained a `case 0x00FE` that folds the standard 8x5 Spectrum-compatible key matrix — reusing `Ports::port[0..7]` verbatim, which `ESPectrum::processKeyboard()` already keeps live every frame regardless of `Config::arch`, so this needed no new key-state plumbing, only the row-select-by-upper-address-byte loop (copied from the existing ULA branch a few lines below it). Two differences from that Spectrum path, both confirmed against FUSE (`peripherals/ula.c`'s `machine->timex` branch) and the TS2068 Technical Manual (Sec. 2.1.13.2's port FEH table): the idle default is the fixed byte `0x5F` (bits 5 and 7 always read 0 — no Issue2/3 quirk on the SCLD), and this port is fully low-byte decoded like 0xF4/0xFF rather than the Spectrum ULA's single-bit (A0==0) partial decode — confirmed against FUSE's `ula_ports_full_decode` table (`machines_periph_timex()`). The write side of port 0xFE (border color, beeper, tape MIC-out) is deliberately **not** implemented yet — a separate concern (audio + border rendering, the latter explicitly still missing per the line below) from keyboard scanning. No new host test: `Ports.cpp` isn't part of the `ESP2068_HOST_TEST` host-testable set (too many ESP-IDF/`VIDEO`/`Tape` dependencies), and the row-select-fold algorithm itself is a verbatim copy of the already-shipped, already-correct Spectrum-side code, not new derived arithmetic — verified instead by a clean `nopsram` build plus a hand trace of the CAPS-SHIFT-alone case against FUSE's confirmed bit table.

**AY-3-8912 sound ports (0xF5/0xF6) are wired now**: `src/Ports.cpp`'s `is2068` switches gained `case 0x00F5`/`case 0x00F6` in both `input()` and `output()`, reusing the existing `AySound` engine verbatim (`AySound::selectedRegister`, `setRegisterData()`, `getRegisterData()`) exactly the way the 128K/+2A/Pentagon path already does a few hundred lines below — no new sound-generation code, only new port decode. Confirmed against FUSE (`machines/tc2068.c`'s `tc2068_ay_registerport_read`/`tc2068_ay_dataport_read`, `machines/machines_periph.c`'s `tc2068_ay_ports` table) and the TS2068 Technical Manual (Sec. 2.1.6, 2.1.13, 4.3): 0xF5 is the register-select port, 0xF6 is data read/write, both fully low-byte decoded like the other TS2068 ports; unlike the plain 128K AY chip (whose data port has no read handler at all), the TS2068's data port can read back any register including 14; and the select port (0xF5) specifically always reads `0xff` while register 14 is selected — `AySound::getRegisterData()` already implements the same I/O-direction-bit gating FUSE's generic read path does, so no new masking logic was needed, only the R14-on-the-select-port special case. A prerequisite fix landed alongside this: `ESPectrum::AY_emu` and the `samplesPerFrame`/`audioOverSampleDivider`/`audioAYDivider`/`audioSampleDivider` audio-mixing constants were never set for `Config::arch == "2068"` (a live divide-by-zero risk the moment a port write pumped the audio buffer) — new `ESP_AUDIO_*_2068` constants (`include/ESPectrum.h`) were derived from `TSTATES_PER_FRAME_2068` (58688, already established in `cpuESP.h`) landing on an exact `31500`Hz output rate, same "perfect division" shape as the existing 48K/Pentagon rows, and `AY_emu = true` unconditionally (matching 128K/+2A/Pentagon — real TS2068 hardware always has the AY chip present, it's not a user-optional flag the way 48K/TK's `Config::AY48` is). **Deliberately not implemented**: joystick-via-AY-register-14 (real TS2068 hardware ANDs inverted joystick state into the data-port read via address bits 8/9 when R14 is selected — there's no Timex-joystick input source anywhere in this codebase yet, and omitting the fold is exactly equivalent to "no joystick connected," not an approximation of it, so nothing was faked). No new host test, same reasoning as the keyboard port: `Ports.cpp` isn't part of the `ESP2068_HOST_TEST` set, and this reuses already-shipped `AySound` logic rather than introducing new derived arithmetic — verified by a clean `nopsram` build instead.

**Not yet done, so don't assume it works:** the OSD menu has no "2068" entry or file picker wired to `DockLoader::loadFromFile()`/`RomLoader`'s fixed paths (Phase 3), AROS autostart itself (only *not incorrectly triggering it* is verified), dual-file/hi-color video modes, mid-frame border-color changes, port 0xFE's write side (border/beeper/tape-out), and Timex-joystick input (register-14 AY reads always report "no buttons pressed") — and nothing has booted on real hardware — even with real ROM/cartridge content now loadable and verified against genuine dumps, everything so far is validated by host-side tests and a clean `nopsram` compile, not by running real software on a real board.

## Upstream facts worth knowing before touching code

- **License: GPL-3.0.** ESPectrum is GPL-3.0-or-later. Anything derived from it — including this port — must stay GPL-3.0 compatible. Don't suggest closed-sourcing, relicensing, or pulling in incompatible-license dependencies.
- **Build system:** PlatformIO, `framework = espidf`, `platform = espressif32@5.4`, `board = pico32`. Two build envs in `platformio.ini`: `psram` (default) and `nopsram`. This port targets `nopsram` — the plan doc's memory budget assumes ~320K usable internal SRAM and explicitly does not need PSRAM.
- **Hardware:** ESP32 dev boards with VGA output (TTGo VGA32-class is the reference board), PS/2 keyboard input, SD card for file/cartridge loading. Same board class works for both Spectrum and TS2068 modes — no new hardware, just new firmware paths.
- **Where the port touches the tree:**
  - `src/CPU.cpp` — has real `Z80Ops::peek8_2068`/`poke8_2068`/`peek16_2068`/`poke16_2068`/`fetchOpcode_2068`/`addressOnBus_2068` now, mirroring `_std`'s shape (`>>13`/`&0x1fff` instead of `>>14`/`&0x3fff`) but as new sibling functions selected via the existing function-pointer dispatch — `_std` itself is untouched and still shared by 48K/128K/Pentagon/TK. (Not `include/Z80_JLS/z80.h` — that file only holds the opcode dispatch table and calls through function pointers defined in `z80operations.h`, implemented in `CPU.cpp`.)
  - `include/MemESP.h` / `src/MemESP.cpp` — **untouched, deliberately.** TS2068 does not use `MemESP::ramCurrent[4]`/`ram[8]`/`rom[5]` at all; it has its own backing store in `SCLD.cpp` (HOME 64K + EXROM 8K + DOCK ≤24K, allocated separately). This coexists with the Spectrum machines rather than replacing their memory model — matches how `Config::arch` already lets one firmware serve multiple machines at runtime.
  - `include/SCLD.h` / `src/SCLD.cpp` — the TS2068 memory model and port 0xF4/0xFF register contract. New files, no upstream equivalent.
  - `include/DockLoader.h` / `src/DockLoader.cpp` — `.DCK` cartridge parsing and LROS-style autostart. New files, no upstream equivalent. Both `SCLD.cpp` and `DockLoader.cpp` also compile host-side under `-DESP2068_HOST_TEST` (see their host tests in `test/host/`) — keep new code in this pair of files free of ESP-IDF-only headers where practical, so that stays true.
  - `include/RomLoader.h` / `src/RomLoader.cpp` — HOME ROM/EXROM loading from a raw binary file. New files, no upstream equivalent, same `-DESP2068_HOST_TEST` convention as `DockLoader.cpp`.
  - `include/SCLDVideo.h` / `src/SCLDVideo.cpp` — byte-to-pixel expansion math for standard/hi-res display modes. New files, no upstream equivalent, no ESP-IDF dependency at all (host-testable without any `-D` flag). Called from `VIDEO::RenderFrame2068()` (`src/Video.cpp`) — see the slice 2 notes above.
  - `src/ESP32Lib/VGA/VGA.h` — has one new `vidmodes[]` row (`SCLD_VGA_MODE_INDEX`) for TS2068's fixed 512-column mode, appended after the existing CRT modes. This is vendored third-party code (`ESP32Lib`, from bitluni); the addition follows its existing pattern (a macro + one array entry) rather than restructuring it.
  - `src/Video.cpp` — `VIDEO::Init()`/`VIDEO::vgataskinit()` special-case `Config::arch == "2068"` to force `SCLD_VGA_MODE_INDEX`, bypassing the Spectrum/TK/Pentagon aspect-ratio/CRT mode-select logic. `VIDEO::Reset()` has an `is2068` branch that sets `VsyncTarget` (frame-pacing, matters for every arch) and returns early, skipping the Spectrum-only border/OSD-position setup that doesn't apply. New method `VIDEO::RenderFrame2068()` does the actual once-per-frame batch render — see the slice 2 notes above. `VIDEO::Draw`/`Draw_Opcode` themselves stay pointed at the existing `&Blank` for TS2068, never retargeted.
  - `src/CPU.cpp`'s `CPU::loop()` — one new branch at the existing per-frame `VIDEO::EndFrame()` call site: `if (Z80Ops::is2068) VIDEO::RenderFrame2068(); else VIDEO::EndFrame();`.
  - `src/Video.cpp` / `ESP32Lib/VGA/VGA.h` / `VGA6Bit.h` — display renderer and the `vidmodes[][17]` VGA timing table. `vga.init()` (which sizes the framebuffer and DMA descriptors) is called exactly once per boot, selected by `Config::arch[0]`; there is no live resolution switch anywhere in ESPectrum. The TS2068 port's resolved design is to boot into one fixed 512-column physical mode always and let the SCLD's port 0xFF video-mode bits pick which renderer fills it (see plan doc's "Display" section) — this avoids needing to invent live mode-switching.
  - `src/Ports.cpp` — I/O port dispatch. Existing Spectrum ports use partial address decoding (e.g. `(address & 0x8002) == 0`); the 2068's ports 0xFF and 0xF4 are fully decoded and the OUT handlers must match the full address, not a bitmask subset.
  - `src/Z80_JLS.cpp` `Z80::check_trdos()` — the "watch PC high byte, repoint a bank pointer, flip a flag" pattern used for TR-DOS autostart. This is the template for LROS/AROS/DOCK cartridge autostart (plan doc, Cartridge loading section).
  - `src/ESPConfig.cpp` (declares `class Config` in `include/ESPConfig.h` — file is `ESPConfig`, not `Config`) / `src/OSDMain.cpp` — machine-selection menu and `Config::arch` string (`"48K"`, `"128K"`, `"+2A"`, `"Pentagon"`, `"TK90X"`, `"TK95"`). A TS2068 entry adds a new arch string and a first-character case in the switches that key off `Config::arch[0]` (e.g. in `Video.cpp`'s mode-select). No existing first character collides with a `2068`-style prefix.

## Conventions for working in this repo

- The plan doc (`TS2068-ESPECTRUM-PORT-PLAN.md`) is the source of truth for the SCLD register contract and memory model. If an implementation detail there turns out to be wrong once code exists, update the doc in the same change — don't let it drift into a stale spec.
- Prefer small, slice-scoped changes that match the four-slice breakdown (MMU/memory core, hi-res renderer + VGA timing, cartridge loader, DOCK/test corpus) over sweeping refactors — the whole point of that split is that the slices share almost no code surface.
- This is embedded C/C++ on a resource-constrained target (no PSRAM in the primary build). Be deliberate about anything that grows static footprint or adds per-access branches on the Z80 peek/poke hot path — the plan doc calls out that the MMU port should be a shift/mask constant change, not a new branch.
- Don't invent write-protect or contention mechanisms that don't exist upstream — ESPectrum has no generalized per-page read-only flag (ROM protection there is a single hardcoded `page == 0` check). This port has its own now: `SCLD::memChunkReadOnly[8]`, gated in `CPU.cpp`'s `poke8_2068`/`poke16_2068`. Use that, don't reinvent it.
