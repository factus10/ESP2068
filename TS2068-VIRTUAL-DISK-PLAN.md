# TS2068 Virtual Disk ‚Äî Design Doc

v0.1. Feeds into [PLAN.md](PLAN.md) the same way [TS2068-ESPECTRUM-PORT-PLAN.md](TS2068-ESPECTRUM-PORT-PLAN.md) fed the original four-slice port. Written after scoping (and rejecting) real FDD3000 hardware emulation and the TS2068's own dead `*device` BASIC syntax ‚Äî see "Why not X" below for both. This is a from-scratch feature: a virtual disk drive, backed by the SD card, that real BASIC programs talk to through a small number of patched dead-end/underspecified entry points ‚Äî three in EXROM, two in HOME ROM (see "Patch summary" below) ‚Äî not an emulation of any specific historical accessory.

## Provenance note

Everything below assumes access to a real, byte-accurate TS2068 EXROM image to patch (not write from scratch) ‚Äî per the project owner: Timex released the TS2068 ROMs to the public domain, so committing the real EXROM (and its patched derivative) into this repo is not the same concern `test/dock/README.md`'s existing provenance policy was written for (that policy predates this clarification and was written for cartridge/ROM dumps of *uncertain* redistribution status). **Open item:** if there's a citable source for the public-domain release (a specific Timex/Sinclair statement, a World of Spectrum licensing page, etc.), worth recording here for any future contributor who re-derives this same question independently ‚Äî not because the project owner's word isn't sufticient to proceed, just so the reasoning survives a change of maintainer. `CLAUDE.md`/`test/dock/README.md` should be updated to reflect this once settled.

## Why not real FDD3000 emulation

Scoped and rejected in favor of this design ‚Äî see `PLAN.md`'s expansion-bank writeup for the parallel case (never-shipped BEU hardware) and this design's own research trail (session history) for FDD3000 specifically: the real FDD3000 contains its *own* Z80 and talks a proprietary nibble-serial framed protocol over port $EF to an onboard WD1771-class controller ‚Äî faithfully emulating that would mean building a second virtual Z80 target and reverse-engineering a bespoke wire protocol, for a payoff (bit-exact behavioral match to one specific 1985 accessory) nobody needs. The FDD3000's *command set* (CAT, MOVE, LOAD, SAVE, VERIFY, MERGE, etc.) is the useful, reusable idea; the wire protocol underneath it is not.

## Why not the `*device` BASIC syntax

`LOAD *"device",...` looks, from the Technical Manual and `docs/ts2068_memory_map.md`, like the officially-designed extension point ‚Äî Timex really did build a SYSCON-table device-dispatch mechanism with a dedicated "disk command handler" vector per accessory (`docs/ts2068_video_and_cartridges.md`'s "Expansion Bank Entry" layout). But it's dead code in the shipped ROM: `NEWDEV` (HOME ROM `$24D2`) parses the `*"device",` syntax correctly and would search that table, but the code path that reaches the search is guarded by an unconditional `JR $2567` ‚Üí *"Error J: Invalid I/O device"*, confirmed at the raw byte level against `2068Home.BIN`, not just the disassembly text. No known ROM revision (including the community bugfix pass documented in `docs/exrom_revision_analysis.md`) ever fixed it. Making it work would mean patching HOME ROM's `NEWDEV`/SYSCON-search logic itself ‚Äî a bigger, more speculative change than any patch this design actually needs (see "Patch summary" below): `LOAD`/`SAVE`/`VERIFY`/`MERGE` route entirely within EXROM, and `CAT`/`CLOSE #` end up needing only small, self-contained HOME ROM redirects, not a resurrection of `NEWDEV`'s dead SYSCON path.

## The real precedent: TS-Pico's own ROM patches

The project owner's own TS-Pico peripheral (`gus-rom-analysis.md` in the local reference library documents Gustavo Pane's shipping TS-PICO ROM) already solves this exact problem on real hardware, and its approach is the template here:

- Patches `W_TAPE` (EXROM `$0068`) and `R_TAPE` (EXROM `$00FC`) ‚Äî the two low-level tape bit-bang entry points SAVE/LOAD/VERIFY/MERGE call into directly (see "How tape I/O actually dispatches" below) ‚Äî each with a 3-byte `JP` to new intercept code. Hex-verified against the real `2068Exrom.BIN`: `W_TAPE` starts `21 E5 00` (`LD HL,$00E5`, exactly 3 bytes), `R_TAPE` starts `14 08 15` (`INC D` / `EX AF,AF'` / `DEC D`, three 1-byte instructions) ‚Äî both patch points overwrite whole instructions cleanly, nothing left half-clobbered.
- A new system variable, `TP_MODE`, decides "SD mode" vs. "tape mode" per call; tape mode falls through and the original routine runs unmodified (their intercept re-executes the clobbered instructions itself before jumping back, since the original bytes are gone).
- Command/device selection rides as a **string prefix inside the ordinary filename argument** (`SAVE "TPI:SDCARD"`, `LOAD "TPI:FILENAME.TAP"`) ‚Äî evaluated as a plain BASIC string expression, no new syntax.
- Critically: `TPNAME`/`TPNLEN` (new system variables at `$5DD3`/`$5DD5`) store a **pointer + length**, not a fixed-size copy ‚Äî because SLVM's own filename handling truncates to 10 characters (the classic Spectrum tape-name limit) before it's usable, and `"TPI:FILENAME.TAP"` is 16 characters, longer than that limit. This is the detail that makes `"D:realistic-filename.tap"`-style commands actually work rather than silently truncating ‚Äî see "Filenames longer than 10 characters" below.
- TS-Pico *also* patches `DOSAVE` in HOME ROM (`$2547`) and adds a HOME-ROM-resident `CALL_BANK` trampoline. Per the analysis doc this looks tied to their own video-mode-aware dispatch refinement, not something the filename-length problem itself requires.

ESP2068's design follows the same load-bearing pattern (patch the tape entry points, prefix-in-filename, pointer+length for the name) but does **not** need to match TS-Pico's port numbers, message framing, or HOME ROM changes ‚Äî confirmed with the project owner that wire-protocol compatibility with real TS-Pico hardware is not a goal.

## How tape I/O actually dispatches (why patching here is sufficient)

Confirmed against the real EXROM disassembly, hex-checked against `2068Exrom.BIN`:

- `SAVE`/`LOAD`/`VERIFY`/`MERGE` all funnel through a shared parser, `SLVM` (EXROM `$01AB`) ‚Äî entered via `CALLBANKS` from HOME ROM's `DOSAVE` (`$2547`), itself reached from the class-11 BASIC dispatch (`PASAVE`/`PALOAD`/`PAVERI`/`PAMERG`) once `NEWDEV`'s `*` check falls through (no `*` present ‚Äî the common case).
- `SLVM` evaluates the filename as an ordinary string expression (`CALL TEM10`), then copies up to 10 characters into a 17-byte (SAVE) or 34-byte (LOAD/VERIFY/MERGE) header buffer at offset `FILENAME` (`+1`).
- Actual tape hardware access happens through **direct, in-bank calls** to `W_TAPE`/`R_TAPE` ‚Äî not through the general channel/stream system (`OPEN#`/`CHANS`), which is a completely separate subsystem used only for `PRINT`/`INPUT`-style character I/O. `SAVE` calls `W_TAPE` twice (once with `A=0` to write the 17-byte header, once with `A=$FF` and the real data pointer/length to write the payload); `LOAD`/`VERIFY` loop calling `R_TAPE` similarly.

This means two patch points ‚Äî `W_TAPE` and `R_TAPE` ‚Äî see every tape-shaped I/O operation SLVM performs, for all four commands, without needing to touch `SLVM`'s own parsing/BASIC-memory-management logic (block-length checks, array replacement, `LINE`/`CODE`/`DATA`/`SCREEN$` suffix parsing) at all. That logic stays real Timex code, untouched.

## Filenames longer than 10 characters

SLVM's own copy-into-header-buffer step is truncating and happens *before* `W_TAPE`/`R_TAPE` are ever called ‚Äî so by the time our patch fires, the filename argument would normally already be chopped to 10 characters if we relied on reading it back out of the header buffer. TS-Pico's fix (a separate pointer+length system variable, captured before the truncating copy) requires intercepting **one step earlier** than `W_TAPE`/`R_TAPE` alone.

**Confirmed: EXROM-only, no HOME ROM patching.** Since `SLVM` itself is EXROM code ‚Äî and this project controls the full EXROM image being loaded, not just two patch points inside an otherwise-untouched original ‚Äî the string-descriptor capture happens *inside* `SLVM`'s own filename-evaluation step, entirely within EXROM.

**The exact hook point ‚Äî hex-verified against `2068Exrom.BIN`, not just the disassembly text.** `TEM10` (SLVM's first cross-bank call, right after entry) is *not* the right spot ‚Äî it only evaluates the filename expression and leaves the result on the calculator/FP stack, nothing usable in registers yet. A *second* cross-bank call, to a HOME ROM routine named `PGPSTR`, is what actually pops the string descriptor into registers, and that's the real interception point:

- `PGPSTR` is invoked via `CALL $0F99` (EXROM's own local `CALL_BANK` trampoline) at **EXROM `$0215`** ‚Äî confirmed byte-for-byte (`CD 99 0F`) walking forward from SLVM's entry at `$01AB`. A clean, whole 3-byte instruction ‚Äî same shape as the `W_TAPE`/`R_TAPE` patch points, nothing to partially clobber.
- It returns to **`$0218`** (`POP IX`). At that point, `BC` = the filename's real length and `DE` = pointer to its character data ‚Äî confirmed two independent ways: the disassembly's own comment ("pop BC, DE, and A from fp stack"), and the very next real instructions, which immediately use `BC` in a "length > 10" bounds check (`LD HL,-10 / DEC BC / ADD HL,BC / INC BC / JR NC,$0231`) before SLVM's own 10-character truncating copy (`$022E` onward) ever runs. This is the full, un-truncated descriptor.

**Patch mechanics:** redirect the `CALL $0F99` at `$0215` to a new intercept. The intercept calls the real `$0F99` first (replicating what it overwrote ‚Äî same technique the `R_TAPE` patch already needs for its own clobbered `INC D`/`EX AF,AF'`/`DEC D`), inspects `(DE)` for the `"D:"` prefix, then either:
- **No `"D:"` prefix:** `RET`s immediately with `BC`/`DE`/`A` completely undisturbed ‚Äî original SLVM flow proceeds byte-for-byte, including its harmless truncated copy into the header buffer.
- **`"D:"` prefix present:** captures the pointer+length into new system variables (mirroring TS-Pico's `TPNAME`/`TPNLEN` pattern) and sets a mode flag, then `RET`s the same way ‚Äî the truncated header-buffer copy still happens but is simply unused for `D:`-prefixed operations, since the separate `W_TAPE`/`R_TAPE` patches check the mode flag instead of that buffer.

Three EXROM patch points total, confirmed: `W_TAPE` (`$0068`), `R_TAPE` (`$00FC`), and this `PGPSTR`-call redirect (`$0215`). Zero HOME ROM changes, consistent with the scope decision above.

Traced *why* TS-Pico's real hardware touches `DOSAVE`/HOME ROM, rather than assuming their approach was load-bearing for this specific problem: their new `$3CE3` trampoline is explicitly "used by multiple callers in the home ROM to invoke EXROM routines" ‚Äî a boot-time TPI-init hook at `$0A02` (into new EXROM code at `$1630`), their general-purpose TPI BIOS (`$1855`, meant to be callable from arbitrary machine-code programs, not just LOAD/SAVE), and a printer-output redirect (`$0A09`). `DOSAVE`'s rewrite is a side effect of TS-Pico adding *several* new HOME-ROM-callable EXROM entry points and wanting one shared, video-mode-aware dispatch helper for all of them ‚Äî not something the filename-length problem itself requires.

The original, unmodified `DOSAVE` already handles both video modes correctly (branching on `VIDMOD` only to pick *how* to reach `SLVM` ‚Äî `$65D0` vs `$FD90`, almost certainly accounting for the second-display-file mode's relocated machine stack near `$F9C0` ‚Äî both paths land in `SLVM` with identical parameters). Once execution reaches `SLVM`, it has no video-mode awareness at all, and neither does anything patched inside it. Since this design only changes control flow *after* `SLVM` is already correctly running ‚Äî not adding a new HOME-ROM-callable entry point ‚Äî there's nothing here that needs `DOSAVE`'s cooperation. Settled: EXROM-only.

## CAT and CLOSE: real, tokenized keywords ‚Äî confirmed, not assumed

Hex-verified against `2068Home.BIN`'s keyword table (`TOKENS`, ROM `$0098`, token = `$A4 +` table index ‚Äî confirmed three independent ways: the raw ASCII+high-bit strings, a `LD B,$CF` load of CAT's own token value at `$25C8`, and the `KEWTBL` class-table indexing arithmetic at `$1A76`) before relying on either:

- **`CAT`** (token `$CF`) is real and tokenizes correctly, but its routine (`$25C8`) is a two-instruction dead-end ‚Äî `LD B,$CF` / falls into a shared tail with the equally-dead `FORMAT`/`MOVE`/`ERASE` stubs ‚Äî that unconditionally throws Report `$12` "INVALID IO DEVICE," without even attempting `NEWDEV`'s SYSCON lookup. Same shape as `LOAD *"device"`: a real keyword, dead dispatch, safe to intercept without touching the tokenizer or syntax table at all.
- **`CLOSE #`** (token `$D4`) is real, tokenizes correctly, and is *not* dead ‚Äî it's live, working channel-close code (`$139F` ‚Üí `$13BE` `CLCHAN`), the same machinery `OPEN #` uses. But the keyword's stored text is literally `"CLOSE #"` (space and `#` included, one single 7-byte string) and its syntax-table entry (`$1A03`: class 6, unconditional `CALL EXPRN`) hard-requires a numeric expression ‚Äî there is no bare, argument-less `CLOSE` representable in the stock ROM's tokenizer or syntax table, and no free token slot exists to add one (`$A5`‚Äì`$FF` is full; TS2068's own six extra keywords already had to overload low control codes to fit).

**Decision:** unmount via `CLOSE #4`, not a bare `CLOSE`. Per the project owner, stream numbers `-3` to `3` are already defined/in use by the system, so `4` is a safe, unused choice. This needs zero tokenizer or syntax-table changes ‚Äî only a dispatch-level intercept recognizing stream `#4` as "unmount the D: container" rather than a real stream-close, reached through `CLOSE #`'s existing, live entry point.

**This means `CAT` and `CLOSE #4` need HOME ROM patches, unlike `LOAD`/`SAVE`/`VERIFY`/`MERGE`.** Both keywords' dispatch entries live entirely in HOME ROM on stock hardware and never reach EXROM at all:

- `CAT` is dispatched via a **class-10/class-5** syntax-table entry (`PACAT`, `$1A16`: `0A 2C 05 C8 25` = class 10, `,`, class 5, `DEFW $25C8`) ‚Äî a completely different shape from `LOAD`/`SAVE`/`VERIFY`/`MERGE`'s **class-11** entries, which route through `NEWDEV`‚Üí`DOSAVE`‚ÜíEXROM's `SLVM`. `CAT`'s routine at `$25C8` (4 bytes: `LD B,$CF` / `JR $25D6`) never touches EXROM ‚Äî it falls into a dead-end tail shared with the equally-unused `FORMAT`/`MOVE`/`ERASE` stubs and unconditionally errors.
- `CLOSE #`'s real, live code (`$139F`‚Üí`$13BE`) is also entirely HOME-ROM-resident, and its default behavior for an unassigned stream number is a harmless no-op ‚Äî meaning stock `CLOSE #4` does nothing today, not an error, just nothing.

**Confirmed design, per the project owner: patch these dead/underspecified entry points directly, exactly the way `W_TAPE`/`R_TAPE`/the `PGPSTR`-call redirect already do in EXROM** ‚Äî put real addresses where the keyword table currently has dead ends, rather than resurrecting `NEWDEV`'s unrelated SYSCON mechanism. Both patches stay small and self-contained, the same size/risk class as the three EXROM patches, not a TS-Pico-style `DOSAVE`/trampoline rewrite:

- **`CAT`** (`$25C8`, 4 bytes: `06 CF 18 0A`) redirects to a small new routine. It doesn't need to reach EXROM/`SLVM` at all ‚Äî `CAT` needs no tape-specific machinery (header buffers, block writing), just "talk to the new virtual-disk port, print the response." HOME ROM already has native character-output/screen-scroll routines available directly, with no cross-bank call needed. `FORMAT`/`MOVE`/`ERASE` are untouched ‚Äî patching only `CAT`'s own 4 bytes leaves the shared tail address (`$25D6`) intact for them.
- **`CLOSE #4`** needs the entry at `$139F` (or the numeric-argument resolution immediately after it) to special-case stream `4` before falling into the real `$140F`/`$13BE` stream-close logic. The exact instruction-level mechanics (where the evaluated stream-number argument actually lives at `$139F`'s entry ‚Äî register, stack, or calculator-stack representation) haven't been hex-traced yet, unlike the `SLVM`/`PGPSTR` hook ‚Äî flagged as a remaining open item, same rigor bar as the filename-capture point.

## Patch summary

Five patch points confirmed so far, three in EXROM and two in HOME ROM ‚Äî all small, self-contained redirects of existing dead-end or underspecified code, none of them a broader architectural rewrite:

| Patch point | ROM | Address | What's there today | What replaces it |
|---|---|---|---|---|
| `W_TAPE` | EXROM | `$0068` | Real tape-write bit-bang entry | Redirect: real tape write (unmodified, if not in D: mode) or SD write |
| `R_TAPE` | EXROM | `$00FC` | Real tape-read bit-bang entry | Redirect: real tape read (unmodified, if not in D: mode) or SD read |
| `PGPSTR` call in `SLVM` | EXROM | `$0215` | `CALL $0F99` (fetches the filename string descriptor) | Redirect: same call, plus a `"D:"` prefix check and pointer+length capture |
| `CAT` | HOME ROM | `$25C8` | 4-byte dead-end (`INVALID IO DEVICE`) | Redirect: SD-directory or mounted-container listing |
| `CLOSE #4` | HOME ROM | `$139F` (exact mechanics TBD) | Live stream-close code; no-op for an unassigned stream | Redirect: unmount, only for stream `4` specifically |

## Command syntax

- `LOAD "D:filename.ext"` ‚Äî mounts `filename.ext` from the SD card as the active container (analogous to inserting a disk). If the extension is `.tap` or `.tzx`, the file is mounted as a multi-block container and **not** immediately loaded ‚Äî subsequent bare `LOAD "progname"` (no `D:` prefix) commands address blocks *within* the mounted container by name. For any other extension, the file's contents load directly into memory (no container semantics ‚Äî one file, one load).
- `LOAD "progname"` (no prefix) ‚Äî loads the named block from the currently-mounted `.tap`/`.tzx` container. Requires a container to already be mounted; behaves like today's tape loading otherwise (this project doesn't currently have a queryable "find block by name" API ‚Äî see `Tape.cpp`'s block-index engine below ‚Äî so this is new logic, built on top of existing, reusable block-parsing/indexing).
- `SAVE "D:filename.ext"` ‚Äî **one-shot**: opens a fresh file, writes exactly the one block being saved, closes it. No append/multi-block-session semantics ‚Äî confirmed with the project owner: if a user wants a multi-program `.tap` file, they assemble it on their PC/Mac, not by issuing several `SAVE "D:same.tap"` commands in a row. `.tap` output reuses `Tape::Save()`'s existing, real block-writing logic (2-byte length + flag + data + checksum) with one adaptation (`MemESP::readbyte()` ‚Üí SCLD equivalent ‚Äî see "Reused vs. new" below). **`.tzx` output is explicitly not implemented** ‚Äî confirmed no `.tzx` write capability exists anywhere in this codebase today (`Tape_TZX.cpp` is read-only), and per the project owner's instruction, this design does not build it fresh. `SAVE "D:filename.tzx"` should error cleanly rather than silently doing something else.
- `VERIFY "D:filename.ext"` / `MERGE "D:filename.ext"` ‚Äî same direct-to-SD-file semantics as `LOAD "D:..."`, one file, no container mount.
- `VERIFY`/`MERGE` without a `D:`-prefixed argument ‚Äî operate against the currently-mounted container only, same restriction as bare `LOAD "progname"`.
- `CLOSE #4` ‚Äî unmounts the currently-mounted `.tap`/`.tzx` container (see "CAT and CLOSE" above for why `#4` and not bare `CLOSE`). No-op if nothing is mounted, matching stock `CLOSE #`'s existing "unused stream slot" no-op behavior at `$139F`.
- `CAT` ‚Äî two modes depending on state, confirmed:
  - No container currently mounted: lists the SD card's current directory ‚Äî filename + size, one per line ‚Äî printed through HOME ROM's own native character-output routine so its existing scroll/pagination behavior applies for free, no new pagination logic needed.
  - A `.tap`/`.tzx` container currently mounted: lists the named blocks *inside* that container ‚Äî name + type (Program/Number array/Character array/Code) + length, one per line, mirroring the classic disk-catalog format rather than inventing a new one.

## Reused vs. new

Confirmed via direct code research this session, not assumed:

- **`FileUtils`'s directory-index machinery** (`DirToFile`/`getDirStats`, the cached-index-file pattern every OSD file picker already uses) is real, architecture-agnostic, and reusable for the "no container mounted" `CAT` listing ‚Äî with one caveat: `DirToFile` itself is a slow, SD-card sort-merge operation coupled to OSD progress-dialog calls, not something to invoke synchronously on every `CAT`. The reusable fast path is the *cached-index-file* read (`fopen`/`fseek`/`fgets` against the already-built, fixed-128-byte-record index) ‚Äî regenerating the index itself should stay an infrequent, background-triggered operation, not something a live `CAT` blocks on by default.
- **`Tape.cpp`/`Tape_TZX.cpp`'s block-parsing/indexing engine** (`TAP_Open`/`TZX_Open`, `CalcTapBlockPos`/`CalcTZXBlockPos`, `TapeListing`) is genuinely pure, architecture-agnostic file/protocol logic ‚Äî no `MemESP`/`SCLD` entanglement in the parsing core itself. Reusable as-is for "mount a `.tap`/`.tzx` and enumerate/seek its blocks." **Name-based block lookup does not exist as a queryable function today** (the closest thing, `Config::tapFPI`'s `tapinfo` index, is off by default, `.tap`-only, program-headers-only, and implemented as a side effect of a 48K-ROM-specific Z80 PC-trap in `Z80_JLS.cpp` that has no TS2068 equivalent) ‚Äî this needs building fresh, but as a thin layer over the existing block-index engine, not a rewrite of it.
- **`Tape::Save()`** writes real, valid `.tap` blocks today. One call site (`MemESP::readbyte(Z80::getRegIX())`) needs to become a TS2068-appropriate read (through `SCLD`'s memory model, or more likely through the new EXROM intercept's own IX-relative reads, since the intercept already has the data pointer/length from the patched `W_TAPE` call ‚Äî TBD during implementation which layer actually needs the substitution).
- **The "CPU port I/O synchronously touches an open SD file" pattern** is established precedent, not new territory ‚Äî the WD1793 controller (`src/disk/wd1793.cpp`) already does real `fseek`/`fread`/`fwrite` against SD-card disk images from inside `Ports.cpp`'s `IN`/`OUT` handlers, and tape loading does the same at byte granularity. A new virtual-disk port is a new, simpler *protocol* on this same primitive, not a new category of interaction for this codebase.
- **New, from scratch:** the patch/intercept code itself (Z80 assembly ‚Äî `SLVM`'s early-capture hook, `W_TAPE`/`R_TAPE` intercepts, the `"D:"` prefix parser, all in EXROM; the `CAT`/`CLOSE #4` redirects, in HOME ROM); the new I/O port(s) and wire protocol between the patched ROM code and the ESP32 host (design TBD ‚Äî not bound to TS-Pico's port numbers or framing); the ESP32-side command handlers (both `CAT` listing modes, name-based block lookup, one-shot SAVE); any new system variables needed to hold the filename pointer/length and mode flag, in TS2068 RAM (neither ROM can hold mutable state itself).
