### ROM loading — done (2026-07-10)

Cross-cutting work, not one of the four named slices — flagged as an open TODO in slice 1's first increment, addressed once a real TS2068 reference library (with genuine ROM dumps) became available locally.

- **[`include/RomLoader.h`](include/RomLoader.h) / [`src/RomLoader.cpp`](src/RomLoader.cpp)** — reads a raw 16K HOME ROM or 8K EXROM binary from a file (plain stdio, no ESP-IDF dependency, host-testable like `DockLoader`) and hands it to a new `SCLD::loadHomeRom()` (copies into the chunks `allocateMemory()` already reserved) or the existing `SCLD::loadExromImage()`. **Deliberately does not reuse** ESPectrum's existing "custom ROM" mechanism for the Spectrum machines (`OSDMain.cpp`'s `updateROM()`) — that's a self-reflashing trick that scans the running OTA partition for a magic-byte placeholder and rewrites the whole firmware image to the other partition; a subagent investigation confirmed it's deeply specific to the Spectrum machines' multi-ROM-variant setup and not a good fit here. `RomLoader` is the much simpler "read a file, copy the bytes" shape instead.

Missing files are not fatal: falls back to the zeroed placeholder HOME ROM / empty-socket EXROM, same as before this increment.

### Slice 3 — done (2026-07-10), first increment

The C++ `.DCK` loader and an LROS-style autostart mechanism — see [`include/DockLoader.h`](include/DockLoader.h) / [`src/DockLoader.cpp`](src/DockLoader.cpp) for the full design writeup (it's substantial enough to live as a code comment, not just here):

- **`.DCK` parsing** is a direct C++ port of `test/dock/dck_format.py`'s already-verified logic, not a re-derivation — same header/chunk-type/image-layout rules. `loadFromBuffer()` validates the *entire* file structurally before touching `SCLD` at all (truncated header or truncated chunk data → returns false, `SCLD`'s cartridge state is left exactly as it was), so a bad file can never leave the machine half-loaded.
- **Autostart is deliberately simpler than real hardware, and that's a documented decision, not a gap.** Research (see below) found the *shape* of the real TS2068 boot protocol — a cartridge has its own short in-ROM header (a chunk-enable byte + a 2-byte jump address), and the real EXROM's init code reads it and hands off via a "GOTO BANK" routine — but not confirmed byte offsets for that header. Rather than guess, `DockLoader` uses the .DCK container's own (fully-spec'd) chunk-presence bits directly: if a DOCK bank's chunk 0 is present, that's treated as an LROS cartridge (which by definition takes total control from address 0, matching where the Z80 already lands after reset), and `hasAutostart()`/`autostartMmuSelect()` expose the chunk mask to page in. `CPU::reset()`'s `"2068"` branch applies it right after `Z80::reset()` sets PC=0, once per reset — no continuous PC-watching (`check_trdos`-style), because LROS only autostarts at boot, not mid-session, unlike TR-DOS. AROS autostart (depends on a working System ROM, which doesn't exist yet) is explicitly not attempted.
- **Research behind that design:** LROS cartridges "contain Z80 machine code in memory chunk 0 and are in total control of the TS2068 hardware including the RESTART implementation," always autorun, "started after initialization... is finished." AROS is mapped at 0x8000, depends on the System ROM, may or may not autorun. Both direct fetches of the primary source (`timexcomputerworld.com`) failed (site unreachable during this session — connection refused); everything above comes from search-result summaries of that same source, cross-referenced across three separate searches. Treat the *shape* as trustworthy, the *exact byte offsets* of the in-ROM header as unconfirmed — which is exactly why they're not implemented.
- **Host test:** `test/host/dockloader_test.cpp`, 15/15 checks passing — loads slice 4's `synthetic.dck`, confirms the autostart mask and chunk content/read-only flags, confirms unload() falls back to the empty socket, confirms a missing-file and a truncated-buffer load both fail cleanly without corrupting a previously-loaded cartridge.
- **Firmware still builds:** `pio run -e nopsram` → see build log; two new files (`DockLoader.h`/`.cpp`) added, `src/CMakeLists.txt`'s glob picks them up automatically like `SCLD.cpp` did.

**Explicitly not done at the time:** AROS autostart; mid-session cartridge launch (would need a real EXROM image with a "launch cartridge" entry point — nothing to watch for yet); confirming the in-ROM LROS/AROS header byte layout against the technical manual (would let a future increment support cartridges whose intended chunk mask differs from "whatever's present in the file," and a real jump address other than 0); wiring `DockLoader::loadFromFile()` up to an actual SD-card file picker (Phase 3, OSD menu).

### Slice 3 — second increment (2026-07-10): real ROM/DOCK end-to-end test, and a real bug it caught

The first increment's autostart design rested on an assumption: "LROS cartridges take total control from address 0, so PC=0 after `Z80::reset()` is already correct — no need to parse the in-ROM header's byte offsets we don't have confirmed." **That assumption was wrong**, and testing against genuine cartridge content (not the fully-synthetic fixture from slice 4) is what caught it.

- **The real cartridge:** `Zebra OS-64.BIN` (16384 bytes, from the local reference library) is a real, documented LROS cartridge — `docs/zebra_os64_analysis.md` describes its header precisely, and a hex dump of the actual file confirms every byte: offset 0 unused (`0x00`), offset 1 cartridge type LROS (`0x01`), offset 2-3 start address `0x0005` (little-endian `05 00`), offset 4 chunk spec `0xFC` (chunks 0-1, low-active). **Bytes 0-4 are pure header metadata, not executable code — the real entry point is a `JP $0D9E` instruction sitting at address `$0005`, not address `$0000`.** Landing the Z80 at PC=0 (this port's original design) would execute those header bytes as garbage instructions instead.
- **The fix:** `DockLoader::autostartEntryPoint()` (new) reads the LROS header's 2-byte start-address field (chunk 0, offset 2-3, little-endian) once chunk 0 is loaded. `CPU::reset()`'s autostart hookup now calls `Z80::setRegPC(DockLoader::autostartEntryPoint())` after paging the cartridge in, instead of relying on `Z80::reset()`'s default PC=0. The chunk-enable mask (`autostartMmuSelect()`) still comes from the `.DCK` container's own chunk-presence bits, not the header's chunk-spec byte — those two agree for both real cartridges tested, and the container bits are the fully-spec'd source already being parsed regardless.
- **The test:** [`test/host/real_cartridge_test.cpp`](test/host/real_cartridge_test.cpp), 18/18 checks, all against genuine cartridge/ROM content (never copied into this repo — read from the local reference library path, in-memory `.DCK` buffers built and discarded within the test, matching `test/dock/README.md`'s provenance policy):
  - **Zebra OS-64** (real LROS): raw header bytes verified first, then loaded through the real `DockLoader` — confirms `autostartEntryPoint() == 0x0005` (the actual bug fix, tested against the exact real value, not a synthetic stand-in) and that the paged-in memory at that offset really is `JP $0D9E`.
  - **eToolkit.ROM** (24576 bytes, a real AROS cartridge — `eToolkit_listing.bas`'s header comment cross-checked byte-for-byte against a hex dump: language=BASIC, type=AROS, start=`$800A`, chunks=`$8F` meaning chunks 4-6) — loaded at chunks 4-6 per its own header, and confirms `hasAutostart()` is correctly **false**, since chunk 0 isn't populated. A meaningful negative case: proves the port doesn't accidentally autostart cartridges it has no business autostarting.
  - **Combined**: real HOME ROM (via `RomLoader`) plus the real Zebra cartridge loaded together — chunk 0 shows HOME ROM before the cartridge is paged in, shows the cartridge's own header once paged in, and reverts cleanly to HOME ROM on eject, undamaged.
- **`dockloader_test.cpp` also updated**: one new check confirms `autostartEntryPoint()` reads the synthetic fixture's actual header bytes (`0x3250`) rather than a hardcoded `0` — a mechanical check that complements the real-cartridge test's semantic one.

This is the clearest example so far in this project of why the "real ROM/DOCK end-to-end test" was worth doing as a distinct step from unit-testing each module in isolation: the bug was invisible to every prior test, because every prior test used either pure logic (no cartridge-shaped content at all) or a synthetic fixture whose header bytes were never meant to resemble a real one.

Still not done: AROS autostart itself (not just detecting it shouldn't fire); mid-session cartridge launch; wiring either loader to a real SD-card file picker (Phase 3).

**Second increment (2026-07-10): renderer integration, with a deliberately different architecture than the first increment predicted.**

The first increment (above) assumed TS2068 would need its own version of the Spectrum machines' interleaved Draw/DrawBorder state machine (`MainScreen`-equivalent functions retargeted by `VIDEO::Draw` as `tstates` crosses scanline boundaries, driven from inside the Z80 core's peek/poke path). Building that would have meant replicating `video_rest`/`coldraw_cnt`/`tstateDraw`/`linedraw_cnt` carry-over bookkeeping and a lazy, port-$FE-triggered border-catch-up mechanism — a lot of intricate, tightly-coupled-to-existing-code machinery whose entire *reason to exist* on the Spectrum side is cycle-accurate rendering under memory contention (mid-instruction attribute changes visible immediately, precise raster-effect timing).

**The SCLD has no contention — so that reason doesn't apply, and a much simpler design works instead.** `VIDEO::Draw`/`Draw_Opcode` stay pointed at the existing `&VIDEO::Blank`/`&Blank_Opcode` (tstates-only, no rendering) for the entire TS2068 session — never retargeted. Instead, [`VIDEO::RenderFrame2068()`](src/Video.cpp) renders the *whole* 512×240 frame in one batch, called once per frame from `CPU::loop()` in place of `VIDEO::EndFrame()` (that call site already exists and already runs exactly once per frame for every machine — no new per-frame hook needed, just an `if (Z80Ops::is2068)` branch on the existing one). This trades mid-frame raster-effect precision (a TS2068 program changing video mode or colors partway down the screen wouldn't show that transition at the right row) for a much smaller, much lower-risk implementation — a reasonable trade for a first increment with no hardware to verify raster-timing edge cases against anyway.

What's covered: standard mode (0x00) and hi-res mode (bit 2 set), full 24-top-border + 192-active + 24-bottom-border frame, FLASH (bit 7, reusing the existing shared `VIDEO::flashing` toggle rather than reimplementing frame-parity tracking), BRIGHT (bit 6, via the existing `zxColor()`/`spectrum_colors[]` table — no new color table needed, TS2068 and Spectrum share the same 8-color-plus-bright palette). Pixel addressing within a display file uses the classic Spectrum "interleaved thirds" layout — confirmed for standard mode against the reference library's documented formula, and additionally hand-verified against known ZX Spectrum addressing facts at several reference points (row 0, row 1, row 8, row 64, and the row-191/col-31 boundary landing exactly on the display file's 6144-byte end) before trusting the transcription; the hi-res mode's dual-file addressing reuses the same formula, which is an assumption (see `SCLDVideo.h`), not independently confirmed.

**Explicitly not done:** dual-file (bit 0) and hi-color (bit 1) modes both currently render as standard mode from DFILE1 — wrong for those specific modes, but shows something recognizable rather than crashing or going blank; hi-color's deferral past v1 was already an open decision, dual-file's is a new, real gap. Border color never changes mid-session (fixed black — port 0xFE isn't wired for TS2068 at all yet, a pre-existing gap from slice 1's scope, not new). No OSD-menu-over-a-running-TS2068-screen compositing (the Spectrum machines' `Draw_OSD169`/`BottomBorder_OSD` mechanism isn't hooked up — the menu likely still works while TS2068 is paused/not yet booted, just not composited live). And, as ever: nothing has been seen on an actual monitor. Everything above is verified by code review, hand-verification of the addressing math, and a clean `nopsram` compile — not by watching a screen.

### Cross-referencing FUSE (2026-07-10): a validation pass that found four real bugs

The project owner pointed at a local checkout of the FUSE emulator (`/Users/david/Downloads/fuse-1.9.0`) — a mature, real, GPL, actively-maintained emulator that has emulated the TS2068 for over a decade (`peripherals/scld.c`/`scld.h`, `peripherals/dck.c`, `machines/ts2068.c`/`tc2068.c`). Its `libspectrum` sibling library was already this project's source for the 60Hz timing constants (slice 1); having the actual emulator source to check the rest of the port against was a substantial upgrade over the TS2068 reference library's summarized docs, which — as this pass demonstrated — were not always right even when they agreed with each other three times over. Four real, confirmed bugs came out of this pass, none of them caught by the 89 host-side checks that existed before it, because none of those checks exercised real Z80 port I/O or real pixel-address arithmetic end to end.

**1. Port address matching (`src/Ports.cpp`) — the most severe.** Slice 1's `is2068` guard matched `switch (address) { case 0x00F4: ... }` — a full 16-bit address match. Real single-byte port I/O (`OUT (n),A` / `IN A,(n)`, `Z80_JLS.cpp`'s `decodeOpcoded3`/`decodeOpcodedb`) puts the Z80 accumulator on the address bus's upper byte, so `LD A,val; OUT ($F4),A` sends `(val<<8)|0xF4` — a 16-bit match only succeeds when `val` happens to be `0`. Confirmed against FUSE's own SCLD port table (`peripherals/scld.c`: `{ 0x00ff, 0x00f4, ... }`, matched as `(port & mask) == value`) before fixing. Fixed to `switch (address & 0xFF)`. This would have made the MMU port almost completely non-functional for any real program writing a genuinely useful (non-zero) value to 0xF4 or 0xFF — invisible to every prior test because all of them called `SCLD::OUT_F4()`/`OUT_FF()` directly, bypassing `Ports.cpp` entirely. `SCLD.h`'s and this doc's earlier "fully decoded... must match the full port address" phrasing (from v0.1) is corrected: full *low-byte* decode, not full 16-bit — see the "two SCLD ports" section above.

**2. Hi-res mode's dual-file interleave order (`SCLDVideo::expandHiresBytePair`).** Shipped as bit-by-bit alternating (DFILE1 bit *i* → output 2*i*, DFILE2 bit *i* → output 2*i*+1). FUSE's `uidisplay_plot16` (`ui/fb/fbdisplay.c`) shows the real arrangement is a **block**, not an interleave: a 16-bit value `(dfile1_byte << 8) | dfile2_byte` expanded MSB-first straight across — i.e. all 8 bits of the DFILE1 byte, then all 8 bits of the DFILE2 byte. Fixed; `SCLDVideo.h`'s "not independently confirmed" caveat is now a confirmed correction instead.

**3. Which DECR value means "hi-res" (`VIDEO::RenderFrame2068`).** The bit-value correction from earlier in this slice (0x04, not v0.1's 0x06) turned out to be half-right: FUSE's `scld.h` shows the DECR byte is read TWO ways at once — `.name.hires` (bit 2 alone, boolean) really is what gates "hi-res category is active," matching the reference library's confirmation — but WITHIN that category, `.mask.scrnmode` (the full 3-bit value, 4 through 7) selects one of **four** distinct sub-modes with different pixel algorithms (`HIRESATTR`=4, `HIRESATTRALTD`=5, `HIRES`=6, `HIRESDOUBLECOL`=7 — see `display.c` lines ~391-412 for each one's exact byte-sourcing). This project's dual-bitmap algorithm is only correct for `HIRES` (0x06) specifically. Fixed the trigger from `videoMode & 0x04` to `videoMode == 0x06`; the other three DECR values now correctly fall into the same "not implemented, render as standard mode" fallback as dual-file/hi-color, rather than silently running the wrong hi-res algorithm. The other three sub-modes are now precisely scoped for whoever implements them next (source their second byte from an attribute area instead of a bitmap area — exact FUSE line references in `SCLDVideo.h`), rather than vaguely "not confirmed."

**4. Standard-mode attribute-row addressing (`Video.cpp`, inline, untested) — the one that had been hiding in plain sight.** `attRowOffset = 0x1800 + row * 32` used the raw *pixel* row (0..191). Attributes are per 8×8 character cell and only change every 8 pixel rows; the correct shape (confirmed against FUSE's `display_attr_start[y] = 6144 + 32*(y/8)`) is `0x1800 + (row >> 3) * 32`. This would have read attribute data from a rapidly-drifting wrong offset almost immediately into any real screen — a standard-mode-breaking bug, not a hi-res edge case. Notably, the *bitmap*-row formula sitting right next to it in the same function was correct (independently hand-verified earlier against known Spectrum addressing facts, and now re-confirmed exactly against FUSE's `display_line_start[]` init loop) — the asymmetry is the lesson: hand-verification without an automated test is exactly how the attribute-row bug got through undetected. Both formulas are now `SCLDVideo::standardBitmapOffset()`/`standardAttributeOffset()` — pure, tested functions (5 checks each, including a same-character-row-shares-an-offset check that directly targets this bug class) — instead of unverified inline arithmetic in `Video.cpp`.

**Also checked and confirmed correct, no fix needed:** the `.DCK` chunk-type model (`peripherals/dck.c`'s `LIBSPECTRUM_DCK_PAGE_NULL/ROM/RAM_EMPTY/RAM` maps 1:1 to this project's `0x00/0x02/0x01/0x03`). **Noted, not fixed:** FUSE's `.DCK` loader supports HOME-bank RAM chunks (preloading HOME RAM content from a cartridge file); `DockLoader` still deliberately skips `BANK_HOME` entirely — a real, low-priority feature gap, not a bug, since this port's HOME RAM already comes from `RomLoader`/real hardware state rather than `.DCK` files.

**Forward note (no fix needed yet — keyboard port 0xFE isn't implemented in this slice):** the project owner flagged that Z80 8-bit port I/O (`OUT (n),A`/`IN A,(n)`) has a lesser-known 16-bit sibling — loading a full 16-bit value into BC and using `OUT (C),r`/`IN r,(C)` — and that real Spectrum/TS2068 keyboard-scan code leans on this for the row-select value. FUSE confirms the mechanism precisely: `peripherals/ula.c`'s port table decodes `0xFE` the same low-byte-only way as the port fix above (`{ 0x00ff, 0x00fe, ula_read, ula_write }`), but `ula_read()` then does `r &= keyboard_read( port >> 8 )` — pulling the row-select mask from the address's *upper* byte and handing it to `keyboard.c`'s `keyboard_read(libspectrum_byte porth)`. So port decode (low byte, mask-based, same shape as the 0xF4/0xFF fix) and the data a read actually needs (upper byte, for 0xFE specifically) are two separate concerns that happen to share one address. `Ports.cpp`'s `is2068`/`input()` switch now carries a comment citing this at the point where the keyboard port will eventually be added, so the upper-byte requirement isn't rediscovered the hard way later. This doesn't change the 0xF4/0xFF fix above — those two ports have no per-read data encoded in the upper byte, so `switch (address & 0xFF)` remains complete for them as-is.

All 6 host test suites re-verified after these fixes (31 SCLD / 15 DockLoader / 26 SCLDVideo / 14 RomLoader / 18 RealCartridge / 13 dck_format = 117 checks total, precisely counted, up from the low-90s before this pass — the new `standardBitmapOffset`/`standardAttributeOffset` checks and the corrected `expandHiresBytePair` checks account for most of the growth). `nopsram` firmware builds clean.

## Open decisions

These are called out in the plan doc too; repeating here because they affect sequencing, not just architecture:

- **TS2068 (60Hz/NTSC) vs TC2068 (50Hz/PAL):** build 60Hz first (it's the target). PAL is a timing-profile variant on the same `Reset()` switch slice 1 already owns — cheap to add after, not worth blocking v1 on.
- **Extended colour mode (010):** recommend deferring past v1. Standard + hi-res cover most native software; this shrinks slice 2's v1 scope without losing much real-world compatibility.
- **Board:** nopsram (TTGo VGA32-class) is the primary target and what the memory budget in the plan doc assumes. Keep one PSRAM board on the bench for double-buffering experiments only — don't let it become a dependency for v1.
- **Repo hosting (new, not in v0.1):** local git history with an `upstream` remote is enough to start (Phase 0). Whether/when to push this to a GitHub fork under your account is a separate call — flag it when you're ready to collaborate with the rest of the team or want CI, since that's the point a fork or a fresh repo with `EremusOne/ESPectrum` credited becomes worth setting up.

## Risk register

- **VGA timing for the new 512-column mode is hand-derived, not copy-paste.** Existing `vidmodes[]` rows top out at 360 active columns and each hand-tunes six APLL clock-divider constants plus porch/sync widths. Budget real monitor time for slice 2, not just a spreadsheet calculation.
- ~~**No existing per-chunk read-only mechanism.**~~ **Resolved by slice 1.** v0.1 of the plan doc assumed ESPectrum's ROM write-protect "generalizes"; verification found it's a single hardcoded `page == 0` check, not a flag array. `SCLD::memChunkReadOnly[8]` is the new one, built fresh as expected, and is what `CPU.cpp`'s `poke8_2068`/`poke16_2068` gate on.
- **DOCK/ROM image provenance.** Real `.DCK` dumps and ROM images may be copyrighted. Fine to use privately for bring-up/testing; be deliberate about what actually gets committed to a repo, especially if it's ever made public. Slice 4 operationalized this: policy is written down in `test/dock/README.md`, and `test/dock/.gitignore` excludes a local-only `real/` directory so local testing doesn't risk an accidental commit. The "ROM loading" increment put this into practice for real: a genuine TS2068 HOME ROM/EXROM (from a local reference library, not this repo) got read and tested against, `git status` was checked before every commit throughout this project specifically to catch this, and no ROM byte ever entered the working tree.
- **GPL-3.0 inheritance.** Any dependency or tool pulled in during this port needs to stay license-compatible with the GPL-3.0 base.
