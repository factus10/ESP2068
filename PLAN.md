### Slice 1 — done (2026-07-10), first increment

Real memory core landed, not just the Phase 1 stub:

- **`src/SCLD.cpp` backing store:** HOME ROM (16K, chunks 0-1) + HOME RAM (48K, chunks 2-7) are real allocated memory (`heap_caps_calloc`/`MALLOC_CAP_8BIT` on-device, matching `MemESP.cpp`'s existing pattern — nothing here needs PSRAM). DOCK (up to 8×8K) and EXROM (a single 8K image, chip-select-mirrored across every chunk that selects it — there's no larger EXROM address space for a second chunk bit to index into) resolve to a shared, zeroed, read-only "empty socket" page until `loadDockChunk()`/`loadExromImage()` populate them (slice 3's job) — `memChunk[]` is never null, even with nothing loaded. `resolveMemChunks()` now does the real HOME/DOCK/EXROM resolution, not a stub.
- **HOME ROM content is a placeholder (zeroed), not a real TS2068 ROM image.** No TS2068 ROM is embedded in this repo — loading a real one (from SD card, matching how the existing Spectrum ROMs' custom-flash procedure works, rather than baking copyrighted bytes into a git-tracked C header) is a separate, not-yet-scoped piece of work. The memory *mechanism* is complete and tested; booting real BASIC is not possible until that lands.
- **`src/CPU.cpp`:** added `Z80Ops::is2068` (alongside `is48`/`is128`/`isPentagon`/`is2a3`, explicitly set in every branch of `CPU::reset()`'s per-arch switch so the flag invariant holds across machine switches) and six `_2068` functions (`fetchOpcode_2068`, `peek8_2068`, `poke8_2068`, `peek16_2068`, `poke16_2068`, `addressOnBus_2068`) mirroring the `_std` versions with `>>13`/`&0x1fff`/`SCLD::memChunk[]`/`memChunkReadOnly[]`, contention always false. **Correction to the plan doc's original framing:** these are new sibling functions selected via the existing `_std`/`_2A3` function-pointer dispatch pattern, not in-place edits to `_std` — editing `_std` directly would have broken 48K/128K/Pentagon/TK, which share it. Also added a `Config::arch == "2068"` branch with real 60Hz timing constants (see below) — this is slice 1 making the trigger string exist in `CPU.cpp`/`Ports.cpp`, not the OSD menu wiring, which stays Phase 3's job as originally scoped.
- **TS2068 timing constants (`include/cpuESP.h`):** `TSTATES_PER_FRAME_2068`=58688, `MICROS_PER_FRAME_2068`=16639 (+125/150-speed variants), `INT_START2068`=0, `INT_END2068`=32. Sourced from [libspectrum](https://github.com/simonowen/libspectrum)'s `timings.c` (the Fuse emulator project ships a `timings_frame_timex_scld_60hz` table for this exact machine: 3,528,000Hz clock, 224 T-states/line, 262 lines/frame = 24 top border + 192 active + 25 bottom border + 21 vsync) — not derived or guessed. Not yet cross-checked against the TS2068 technical/service manual or real hardware; verify at bring-up.
- **`src/Ports.cpp`:** `Z80Ops::is2068` guard added at the top of `input()`/`output()`, fully decoding `address == 0x00F4`/`0x00FF` before any of the Spectrum's partial-decode branches run. This isn't cosmetic: 0xF4 is an *even* address, so without the guard it would fall into the ULA-port branch (`(address & 0x0001) == 0`) and never reach the SCLD at all. Other TS2068 ports (keyboard, AY, etc.) return a stub `0xff`/no-op — out of scope for "memory core."
- **`src/ESPectrum.cpp`:** `SCLD::allocateMemory()` added next to `MemESP::Init()` (called once at boot, not per-reset), `SCLD::reset()` added next to both `MemESP::Reset()` call sites (boot and user-triggered reset).
- **Test: `test/host/scld_test.cpp`** — standalone, host-g++-compilable (no ESP-IDF dependency; `SCLD.cpp` takes a `calloc()` path instead of `heap_caps_calloc()` when `ESP2068_HOST_TEST` is defined, so the test exercises the real firmware code, not a reimplementation). 30 checks, all passing: reset state, all-HOME resolution with correct read-only flags, HOME RAM read/write survives a DOCK round-trip, populated DOCK RAM vs DOCK ROM chunks, EXROM mirroring across multiple chunks, port read-back, and a write-protect check against the same flag `CPU.cpp`'s `poke8_2068`/`poke16_2068` gate on. Build/run: `g++ -std=c++17 -Wall -I include -DESP2068_HOST_TEST test/host/scld_test.cpp src/SCLD.cpp -o /tmp/scld_test && /tmp/scld_test`.
- **Firmware still builds:** `pio run -e nopsram` → `SUCCESS`, RAM 9.7% (31704/327680 B, up from the Phase 0 baseline's 7.1% — the increase is the new static 8K "empty socket" page plus small bookkeeping arrays, not the heap-allocated HOME RAM/ROM which doesn't show in this report), Flash 70.9% (1393953/1966080 B). No errors.

**Explicitly not done in this increment** (tracked so it isn't mistaken for finished): a real TS2068 ROM image and a loader for it; the second write-protect call site flagged in Phase 1 (`MemESP::writebyte`/`writeword`) — turned out not to matter yet, since TS2068 uses its own backing store entirely separate from `MemESP::ram[]`/`rom[]` and doesn't go through `Snapshot.cpp`/`Tape.cpp`'s `.SNA`/`.TAP` paths; it only becomes relevant if something later decides to reuse those paths for TS2068 snapshots. ### Slice 4 — done (2026-07-10), first increment

Offline tooling and a synthetic fixture, not a real image corpus — see [`test/dock/README.md`](test/dock/README.md) for the full writeup:

- **`.DCK` container format nailed down** via web research rather than guessed (it wasn't in either plan doc at byte level): no magic number, a 9-byte header per bank (bank ID + 8 chunk-type bytes), chunk type is a 2-bit field (D0 = writable, D1 = image-present-in-file), 8192-byte images follow the header for present chunks, repeat for additional banks. Sourced from the "Spectrum FAQ - File Formats" reference (the format as originally used by the Warajevo emulator), corroborated against two independent copies of the same text. **Not yet cross-checked against a real `.dck` file** — nobody on this project has one yet; do that first and fix the format doc/parser together if anything's off.
- **`test/dock/dck_format.py`** — parser/writer, raises on truncated headers/chunk data rather than misreading them. **`test/dock/test_dck_format.py`** — round-trip + error-handling checks, 13/13 passing. **`test/dock/dck_inspect.py`** — CLI to print any `.dck`'s bank/chunk structure without interpreting chunk contents.
- **`test/dock/make_synthetic_dck.py`** → `test/dock/fixtures/synthetic.dck` — one DOCK bank, a RAM chunk and a ROM chunk filled with clearly-synthetic patterns (a marker string, a byte ramp), not real code. Proves the container format round-trips; proves nothing about running real software, since there isn't any in it.
- **Provenance policy written down** (`test/dock/README.md`, also `test/dock/.gitignore`'s `real/` pattern): no real cartridge/ROM dumps in this repo without confirmed redistribution rights, real dumps for local testing go in a gitignored `test/dock/real/`, prefer homebrew/public-domain fixtures for anything committed.

**Explicitly not done:** any real `.DCK`/ROM image (by design — see provenance policy), per-mode display test cards for slice 2 (a different, not-yet-started shape of fixture), the actual C++ loader that will consume this format in the firmware (slice 3).

### Slice 3 — done (2026-07-10), first increment

The C++ `.DCK` loader and an LROS-style autostart mechanism — see [`include/DockLoader.h`](include/DockLoader.h) / [`src/DockLoader.cpp`](src/DockLoader.cpp) for the full design writeup (it's substantial enough to live as a code comment, not just here):

- **`.DCK` parsing** is a direct C++ port of `test/dock/dck_format.py`'s already-verified logic, not a re-derivation — same header/chunk-type/image-layout rules. `loadFromBuffer()` validates the *entire* file structurally before touching `SCLD` at all (truncated header or truncated chunk data → returns false, `SCLD`'s cartridge state is left exactly as it was), so a bad file can never leave the machine half-loaded.
- **Autostart is deliberately simpler than real hardware, and that's a documented decision, not a gap.** Research (see below) found the *shape* of the real TS2068 boot protocol — a cartridge has its own short in-ROM header (a chunk-enable byte + a 2-byte jump address), and the real EXROM's init code reads it and hands off via a "GOTO BANK" routine — but not confirmed byte offsets for that header. Rather than guess, `DockLoader` uses the .DCK container's own (fully-spec'd) chunk-presence bits directly: if a DOCK bank's chunk 0 is present, that's treated as an LROS cartridge (which by definition takes total control from address 0, matching where the Z80 already lands after reset), and `hasAutostart()`/`autostartMmuSelect()` expose the chunk mask to page in. `CPU::reset()`'s `"2068"` branch applies it right after `Z80::reset()` sets PC=0, once per reset — no continuous PC-watching (`check_trdos`-style), because LROS only autostarts at boot, not mid-session, unlike TR-DOS. AROS autostart (depends on a working System ROM, which doesn't exist yet) is explicitly not attempted.
- **Research behind that design:** LROS cartridges "contain Z80 machine code in memory chunk 0 and are in total control of the TS2068 hardware including the RESTART implementation," always autorun, "started after initialization... is finished." AROS is mapped at 0x8000, depends on the System ROM, may or may not autorun. Both direct fetches of the primary source (`timexcomputerworld.com`) failed (site unreachable during this session — connection refused); everything above comes from search-result summaries of that same source, cross-referenced across three separate searches. Treat the *shape* as trustworthy, the *exact byte offsets* of the in-ROM header as unconfirmed — which is exactly why they're not implemented.
- **Host test:** `test/host/dockloader_test.cpp`, 15/15 checks passing — loads slice 4's `synthetic.dck`, confirms the autostart mask and chunk content/read-only flags, confirms unload() falls back to the empty socket, confirms a missing-file and a truncated-buffer load both fail cleanly without corrupting a previously-loaded cartridge.
- **Firmware still builds:** `pio run -e nopsram` → see build log; two new files (`DockLoader.h`/`.cpp`) added, `src/CMakeLists.txt`'s glob picks them up automatically like `SCLD.cpp` did.

**Explicitly not done:** AROS autostart; mid-session cartridge launch (would need a real EXROM image with a "launch cartridge" entry point — nothing to watch for yet); confirming the in-ROM LROS/AROS header byte layout against the technical manual (would let a future increment support cartridges whose intended chunk mask differs from "whatever's present in the file," and a real jump address other than 0); wiring `DockLoader::loadFromFile()` up to an actual SD-card file picker (Phase 3, OSD menu).

### Slice 2 — done (2026-07-10), first increment

A real TS2068 reference library became available partway through this slice (`/Users/david/Documents/Projects/TS2068 Ref Library/`) — real ROM disassemblies and cross-referenced documentation, a large step up from the earlier web-search-only sourcing. First and most important thing it did: **found a real error** in this project's own v0.1 plan doc, before any renderer code was built around the wrong value — see "Port 0xFF" in `TS2068-ESPECTRUM-PORT-PLAN.md` for the correction (hi-res/64-column mode is DECR bit 2 alone, `0x04`, not bits `110`/`0x06`; confirmed by three independent files in the reference library). `SCLD.h`'s `videoMode` comment updated to match.

What landed this increment:

- **[`include/SCLDVideo.h`](include/SCLDVideo.h) / [`src/SCLDVideo.cpp`](src/SCLDVideo.cpp)** — the byte-to-pixel expansion core, deliberately factored out as pure, host-testable functions rather than written inline into `Video.cpp`'s renderer machinery (see below for why that's *not* done yet): `expandStandardByte()` (one bitmap byte + one attribute byte → 16 pixel-doubled output pixels, bit 7 = leftmost, confirmed against the reference library's pixel-address formula comment), `expandHiresBytePair()` (one byte from each display file → 16 interleaved output pixels, DFILE1 on even columns / DFILE2 on odd), and `hiresInkPaperColors()` (decodes the ink/paper pair — turns out to reduce to `ink = field, paper = field ^ 7`, a clean pattern that held for all 8 entries in the plan doc's table, a good internal-consistency signal that *that* part of v0.1 was right even though the video-mode bits weren't). Host test: `test/host/scldvideo_test.cpp`, 15/15 checks.
- **New `vidmodes[]` row** (`src/ESP32Lib/VGA/VGA.h`, `VgaMode_512x480_60_TS2068`, index 29 / `SCLD_VGA_MODE_INDEX`) — the physical VGA timing for the fixed 512-active-column mode this port always boots into (per Phase 1's resolved design). Computed with the *actual* APLL clock-divider algorithm from `tools/VideoModeTool.py` (Jean Thomas' ESP32-APLL-cal, the same math upstream's existing rows were derived with) — not hand-waved. Horizontal porches scaled ×1.6 from the proven `VgaMode_320x240_60_TK` (320×1.6=512 exactly, so this mode shares that mode's ~31425Hz horizontal frequency — same line rate as something already known to sync real monitors on this driver, just packing more pixels per line); vertical timing copied unchanged from the same mode (480/vDiv=2 → 240-line framebuffer, close to TS2068's real 24+192+24 border/active/border shape); vertical frequency targets that mode's proven 59.856887Hz rather than the separately-derived ~60.11Hz from `TSTATES_PER_FRAME_2068` (different clock domain — see the code comment). REV1 params hit the target frequency at 0.0000% error. **Still not bench-verified on a real monitor** — this is a mathematically self-consistent starting point for that bench pass, exactly the risk the register below already flagged, not a substitute for it.
- **`src/Video.cpp` wiring**: `VIDEO::Init()` and `VIDEO::vgataskinit()` now recognize `Config::arch == "2068"` and force `Mode = SCLD_VGA_MODE_INDEX`, bypassing the aspect-ratio/scanlines/CRT selection that doesn't apply to this machine. This is the one piece of this increment that's genuinely mechanical/low-risk (an early-exit branch, not new rendering logic).

**Explicitly not done, and this is the substantial remaining part of slice 2:** nothing wires `SCLDVideo`'s expansion functions into an actual live renderer yet. `VIDEO::Draw`/`VIDEO::Draw_Opcode` are self-retargeting function pointers — the existing Spectrum renderers (`MainScreen`, `MainScreen_2A3`, etc.) repoint `Draw` to different functions as `tstates` crosses scanline/frame boundaries, which is how ESPectrum achieves cycle-accurate interleaved rendering without a separate video thread (this is a real architectural discovery from reading `Video.cpp`, not something documented anywhere going in). TS2068 needs its own version of that state machine — `MainScreen_2068`-equivalent functions that read from `SCLD::memChunk`-resolved DFILE bytes, call `SCLDVideo::expandStandardByte`/`expandHiresBytePair`, and write into `vga.frameBuffer` at the right scanline/column offsets, all while being retargeted by tstates the same way the existing renderers are. That's a large, tightly-coupled-to-existing-code piece of work, and getting it wrong is hard to detect without actually watching a screen — attempting it without hardware to verify against felt like the wrong tradeoff for this increment, versus landing the algorithmic core solid and tested first. Also not done: extended-colour/hi-color mode (deferred past v1 per the open decisions below, unchanged); the real hi-res DFILE1/DFILE2-interleave order remains not independently hardware-confirmed (see `SCLDVideo.h`'s comment — it matches the original plan doc's description and is internally consistent, but wasn't confirmed by the new reference library either).

## Open decisions

These are called out in the plan doc too; repeating here because they affect sequencing, not just architecture:

- **TS2068 (60Hz/NTSC) vs TC2068 (50Hz/PAL):** build 60Hz first (it's the target). PAL is a timing-profile variant on the same `Reset()` switch slice 1 already owns — cheap to add after, not worth blocking v1 on.
- **Extended colour mode (010):** recommend deferring past v1. Standard + hi-res cover most native software; this shrinks slice 2's v1 scope without losing much real-world compatibility.
- **Board:** nopsram (TTGo VGA32-class) is the primary target and what the memory budget in the plan doc assumes. Keep one PSRAM board on the bench for double-buffering experiments only — don't let it become a dependency for v1.
- **Repo hosting (new, not in v0.1):** local git history with an `upstream` remote is enough to start (Phase 0). Whether/when to push this to a GitHub fork under your account is a separate call — flag it when you're ready to collaborate with the rest of the team or want CI, since that's the point a fork or a fresh repo with `EremusOne/ESPectrum` credited becomes worth setting up.

## Risk register

- **VGA timing for the new 512-column mode is hand-derived, not copy-paste.** Existing `vidmodes[]` rows top out at 360 active columns and each hand-tunes six APLL clock-divider constants plus porch/sync widths. Budget real monitor time for slice 2, not just a spreadsheet calculation.
- ~~**No existing per-chunk read-only mechanism.**~~ **Resolved by slice 1.** v0.1 of the plan doc assumed ESPectrum's ROM write-protect "generalizes"; verification found it's a single hardcoded `page == 0` check, not a flag array. `SCLD::memChunkReadOnly[8]` is the new one, built fresh as expected, and is what `CPU.cpp`'s `poke8_2068`/`poke16_2068` gate on.
- **DOCK/ROM image provenance.** Real `.DCK` dumps and ROM images may be copyrighted. Fine to use privately for bring-up/testing; be deliberate about what actually gets committed to a repo, especially if it's ever made public. Slice 4 operationalized this: policy is written down in `test/dock/README.md`, and `test/dock/.gitignore` excludes a local-only `real/` directory so local testing doesn't risk an accidental commit.
- **GPL-3.0 inheritance.** Any dependency or tool pulled in during this port needs to stay license-compatible with the GPL-3.0 base.
