### ROM loading — done (2026-07-10)

Cross-cutting work, not one of the four named slices — flagged as an open TODO in slice 1's first increment, addressed once a real TS2068 reference library (with genuine ROM dumps) became available locally.

- **[`include/RomLoader.h`](include/RomLoader.h) / [`src/RomLoader.cpp`](src/RomLoader.cpp)** — reads a raw 16K HOME ROM or 8K EXROM binary from a file (plain stdio, no ESP-IDF dependency, host-testable like `DockLoader`) and hands it to a new `SCLD::loadHomeRom()` (copies into the chunks `allocateMemory()` already reserved) or the existing `SCLD::loadExromImage()`. **Deliberately does not reuse** ESPectrum's existing "custom ROM" mechanism for the Spectrum machines (`OSDMain.cpp`'s `updateROM()`) — that's a self-reflashing trick that scans the running OTA partition for a magic-byte placeholder and rewrites the whole firmware image to the other partition; a subagent investigation confirmed it's deeply specific to the Spectrum machines' multi-ROM-variant setup and not a good fit here. `RomLoader` is the much simpler "read a file, copy the bytes" shape instead.

Missing files are not fatal: falls back to the zeroed placeholder HOME ROM / empty-socket EXROM, same as before this increment.

### Slice 3 — done (2026-07-10), first increment

The C++ `.DCK` loader and an LROS-style autostart mechanism — see [`include/DockLoader.h`](include/DockLoader.h) / [`src/DockLoader.cpp`](src/DockLoader.cpp) for the full design writeup (it's substantial enough to live as a code comment, not just here):

- **`.DCK` parsing** is a direct C++ port of `test/dock/dck_format.py`'s already-verified logic, not a re-derivation — same header/chunk-type/image-layout rules. `loadFromBuffer()` validates the *entire* file structurally before touching `SCLD` at all (truncated header or truncated chunk data → returns false, `SCLD`'s cartridge state is left exactly as it was), so a bad file can never leave the machine half-loaded.
- **Autostart is deliberately simpler than real hardware, and that's a documented decision, not a gap.** Research (see below) found the *shape* of the real TS2068 boot protocol — a cartridge has its own short in-ROM header (a chunk-enable byte + a 2-byte jump address), and the real EXROM's init code reads it and hands off via a "GOTO BANK" routine — but not confirmed byte offsets for that header. Rather than guess, `DockLoader` uses the .DCK container's own (fully-spec'd) chunk-presence bits directly: if a DOCK bank's chunk 0 is present, that's treated as an LROS cartridge (which by definition takes total control from address 0, matching where the Z80 already lands after reset), and `hasAutostart()`/`autostartMmuSelect()` expose the chunk mask to page in. `CPU::reset()`'s `"2068"` branch applies it right after `Z80::reset()` sets PC=0, once per reset — no continuous PC-watching (`check_trdos`-style), because LROS only autostarts at boot, not mid-session, unlike TR-DOS. AROS autostart (depends on a working System ROM, which doesn't exist yet) is explicitly not attempted.
- **Research behind that design:** LROS cartridges "contain Z80 machine code in memory chunk 0 and are in total control of the TS2068 hardware including the RESTART implementation," always autorun, "started after initialization... is finished." AROS is mapped at 0x8000, depends on the System ROM, may or may not autorun. Both direct fetches of the primary source (`timexcomputerworld.com`) failed (site unreachable during this session — connection refused); everything above comes from search-result summaries of that same source, cross-referenced across three separate searches. Treat the *shape* as trustworthy, the *exact byte offsets* of the in-ROM header as unconfirmed — which is exactly why they're not implemented.
- **Host test:** `test/host/dockloader_test.cpp`, 15/15 checks passing — loads slice 4's `synthetic.dck`, confirms the autostart mask and chunk content/read-only flags, confirms unload() falls back to the empty socket, confirms a missing-file and a truncated-buffer load both fail cleanly without corrupting a previously-loaded cartridge.
- **Firmware still builds:** `pio run -e nopsram` → see build log; two new files (`DockLoader.h`/`.cpp`) added, `src/CMakeLists.txt`'s glob picks them up automatically like `SCLD.cpp` did.

**Explicitly not done at the time:** AROS autostart; mid-session cartridge launch (would need a real EXROM image with a "launch cartridge" entry point — nothing to watch for yet); confirming the in-ROM LROS/AROS header byte layout against the technical manual (would let a future increment support cartridges whose intended chunk mask differs from "whatever's present in the file," and a real jump address other than 0); wiring `DockLoader::loadFromFile()` up to an actual SD-card file picker (Phase 3, OSD menu).

### Slice 3 — second increment (2026-07-10): real ROM/DOCK end-to-end test, and a real bug it caught

The first increment's autostart design rested on an assumption: "LROS cartridges take total control from address 0, so PC=0 after `Z80::reset()` is already correct — no need to parse the in-ROM header's byte offsets we don't have confirmed." **That assumption was wrong**, and testing against genuine cartridge content (not the fully-synthetic fixture from slice 4) is what caught it.

- **The real cartridge:** `Zebra OS-64.BIN` (16384 bytes, from the local reference library) is a real, documented LROS cartridge — `docs/zebra_os64_analysis.md` describes its header precisely, and a hex dump of the actual file confirms every byte: offset 0 unused (`0x00`), offset 1 cartridge type LROS (`0x01`), offset 2-3 start address `0x0005` (little-endian `05 00`), offset 4 chunk spec `0xFC` (chunks 0-1, low-active). **Bytes 0-4 are pure header metadata, not executable code — the real entry point is a `JP $0D9E` instruction sitting at address `$0005`, not address `$0000`.** Landing the Z80 at PC=0 (this port's original design) would execute those header bytes as garbage instructions instead.
- **The fix:** `DockLoader::autostartEntryPoint()` (new) reads the LROS header's 2-byte start-address field (chunk 0, offset 2-3, little-endian) once chunk 0 is loaded. `CPU::reset()`'s autostart hookup now calls `Z80::setRegPC(DockLoader::autostartEntryPoint())` after paging the cartridge in, instead of relying on `Z80::reset()`'s default PC=0. The chunk-enable mask (`autostartMmuSelect()`) still comes from the `.DCK` container's own chunk-presence bits, not the header's chunk-spec byte — those two agree for both real cartridges tested, and the container bits are the fully-spec'd source already being parsed regardless.
- **The test:** [`test/host/real_cartridge_test.cpp`](test/host/real_cartridge_test.cpp), 18/18 checks, all against genuine cartridge/ROM content (never copied into this repo — read from the local reference library path, in-memory `.DCK` buffers built and discarded within the test, matching `test/dock/README.md`'s provenance policy):
  - **Zebra OS-64** (real LROS): raw header bytes verified first, then loaded through the real `DockLoader` — confirms `autostartEntryPoint() == 0x0005` (the actual bug fix, tested against the exact real value, not a synthetic stand-in) and that the paged-in memory at that offset really is `JP $0D9E`.
  - **eToolkit.ROM** (24576 bytes, a real AROS cartridge — `eToolkit_listing.bas`'s header comment cross-checked byte-for-byte against a hex dump: language=BASIC, type=AROS, start=`$800A`, chunks=`$8F` meaning chunks 4-6) — loaded at chunks 4-6 per its own header, and confirms `hasAutostart()` is correctly **false**, since chunk 0 isn't populated. A meaningful negative case: proves the port doesn't accidentally autostart cartridges it has no business autostarting.
  - **Combined**: real HOME ROM (via `RomLoader`) plus the real Zebra cartridge loaded together — chunk 0 shows HOME ROM before the cartridge is paged in, shows the cartridge's own header once paged in, and reverts cleanly to HOME ROM on eject, undamaged.
- **`dockloader_test.cpp` also updated**: one new check confirms `autostartEntryPoint()` reads the synthetic fixture's actual header bytes (`0x3250`) rather than a hardcoded `0` — a mechanical check that complements the real-cartridge test's semantic one.

This is the clearest example so far in this project of why the "real ROM/DOCK end-to-end test" was worth doing as a distinct step from unit-testing each module in isolation: the bug was invisible to every prior test, because every prior test used either pure logic (no cartridge-shaped content at all) or a synthetic fixture whose header bytes were never meant to resemble a real one.

Still not done: AROS autostart itself (not just detecting it shouldn't fire); mid-session cartridge launch; wiring either loader to a real SD-card file picker (Phase 3).

**Second increment (2026-07-10): renderer integration, with a deliberately different architecture than the first increment predicted.**

The first increment (above) assumed TS2068 would need its own version of the Spectrum machines' interleaved Draw/DrawBorder state machine (`MainScreen`-equivalent functions retargeted by `VIDEO::Draw` as `tstates` crosses scanline boundaries, driven from inside the Z80 core's peek/poke path). Building that would have meant replicating `video_rest`/`coldraw_cnt`/`tstateDraw`/`linedraw_cnt` carry-over bookkeeping and a lazy, port-$FE-triggered border-catch-up mechanism — a lot of intricate, tightly-coupled-to-existing-code machinery whose entire *reason to exist* on the Spectrum side is cycle-accurate rendering under memory contention (mid-instruction attribute changes visible immediately, precise raster-effect timing).

**The SCLD has no contention — so that reason doesn't apply, and a much simpler design works instead.** `VIDEO::Draw`/`Draw_Opcode` stay pointed at the existing `&VIDEO::Blank`/`&Blank_Opcode` (tstates-only, no rendering) for the entire TS2068 session — never retargeted. Instead, [`VIDEO::RenderFrame2068()`](src/Video.cpp) renders the *whole* 512×240 frame in one batch, called once per frame from `CPU::loop()` in place of `VIDEO::EndFrame()` (that call site already exists and already runs exactly once per frame for every machine — no new per-frame hook needed, just an `if (Z80Ops::is2068)` branch on the existing one). This trades mid-frame raster-effect precision (a TS2068 program changing video mode or colors partway down the screen wouldn't show that transition at the right row) for a much smaller, much lower-risk implementation — a reasonable trade for a first increment with no hardware to verify raster-timing edge cases against anyway.

What's covered: standard mode (0x00) and hi-res mode (bit 2 set), full 24-top-border + 192-active + 24-bottom-border frame, FLASH (bit 7, reusing the existing shared `VIDEO::flashing` toggle rather than reimplementing frame-parity tracking), BRIGHT (bit 6, via the existing `zxColor()`/`spectrum_colors[]` table — no new color table needed, TS2068 and Spectrum share the same 8-color-plus-bright palette). Pixel addressing within a display file uses the classic Spectrum "interleaved thirds" layout — confirmed for standard mode against the reference library's documented formula, and additionally hand-verified against known ZX Spectrum addressing facts at several reference points (row 0, row 1, row 8, row 64, and the row-191/col-31 boundary landing exactly on the display file's 6144-byte end) before trusting the transcription; the hi-res mode's dual-file addressing reuses the same formula, which is an assumption (see `SCLDVideo.h`), not independently confirmed.

**Explicitly not done:** dual-file (bit 0) and hi-color (bit 1) modes both currently render as standard mode from DFILE1 — wrong for those specific modes, but shows something recognizable rather than crashing or going blank; hi-color's deferral past v1 was already an open decision, dual-file's is a new, real gap. Border color never changes mid-session (fixed black — port 0xFE isn't wired for TS2068 at all yet, a pre-existing gap from slice 1's scope, not new). No OSD-menu-over-a-running-TS2068-screen compositing (the Spectrum machines' `Draw_OSD169`/`BottomBorder_OSD` mechanism isn't hooked up — the menu likely still works while TS2068 is paused/not yet booted, just not composited live). And, as ever: nothing has been seen on an actual monitor. Everything above is verified by code review, hand-verification of the addressing math, and a clean `nopsram` compile — not by watching a screen.

Closest of the four slices to done, still the one with the least real-world verification given how much of it is about what a screen looks like.

### Cross-referencing FUSE (2026-07-10): a validation pass that found four real bugs

The project owner pointed at a local checkout of the FUSE emulator (`/Users/david/Downloads/fuse-1.9.0`) — a mature, real, GPL, actively-maintained emulator that has emulated the TS2068 for over a decade (`peripherals/scld.c`/`scld.h`, `peripherals/dck.c`, `machines/ts2068.c`/`tc2068.c`). Its `libspectrum` sibling library was already this project's source for the 60Hz timing constants (slice 1); having the actual emulator source to check the rest of the port against was a substantial upgrade over the TS2068 reference library's summarized docs, which — as this pass demonstrated — were not always right even when they agreed with each other three times over. Four real, confirmed bugs came out of this pass, none of them caught by the 89 host-side checks that existed before it, because none of those checks exercised real Z80 port I/O or real pixel-address arithmetic end to end.

**1. Port address matching (`src/Ports.cpp`) — the most severe.** Slice 1's `is2068` guard matched `switch (address) { case 0x00F4: ... }` — a full 16-bit address match. Real single-byte port I/O (`OUT (n),A` / `IN A,(n)`, `Z80_JLS.cpp`'s `decodeOpcoded3`/`decodeOpcodedb`) puts the Z80 accumulator on the address bus's upper byte, so `LD A,val; OUT ($F4),A` sends `(val<<8)|0xF4` — a 16-bit match only succeeds when `val` happens to be `0`. Confirmed against FUSE's own SCLD port table (`peripherals/scld.c`: `{ 0x00ff, 0x00f4, ... }`, matched as `(port & mask) == value`) before fixing. Fixed to `switch (address & 0xFF)`. This would have made the MMU port almost completely non-functional for any real program writing a genuinely useful (non-zero) value to 0xF4 or 0xFF — invisible to every prior test because all of them called `SCLD::OUT_F4()`/`OUT_FF()` directly, bypassing `Ports.cpp` entirely. `SCLD.h`'s and this doc's earlier "fully decoded... must match the full port address" phrasing (from v0.1) is corrected: full *low-byte* decode, not full 16-bit — see the "two SCLD ports" section above.

**2. Hi-res mode's dual-file interleave order (`SCLDVideo::expandHiresBytePair`).** Shipped as bit-by-bit alternating (DFILE1 bit *i* → output 2*i*, DFILE2 bit *i* → output 2*i*+1). FUSE's `uidisplay_plot16` (`ui/fb/fbdisplay.c`) shows the real arrangement is a **block**, not an interleave: a 16-bit value `(dfile1_byte << 8) | dfile2_byte` expanded MSB-first straight across — i.e. all 8 bits of the DFILE1 byte, then all 8 bits of the DFILE2 byte. Fixed; `SCLDVideo.h`'s "not independently confirmed" caveat is now a confirmed correction instead.

**3. Which DECR value means "hi-res" (`VIDEO::RenderFrame2068`).** The bit-value correction from earlier in this slice (0x04, not v0.1's 0x06) turned out to be half-right: FUSE's `scld.h` shows the DECR byte is read TWO ways at once — `.name.hires` (bit 2 alone, boolean) really is what gates "hi-res category is active," matching the reference library's confirmation — but WITHIN that category, `.mask.scrnmode` (the full 3-bit value, 4 through 7) selects one of **four** distinct sub-modes with different pixel algorithms (`HIRESATTR`=4, `HIRESATTRALTD`=5, `HIRES`=6, `HIRESDOUBLECOL`=7 — see `display.c` lines ~391-412 for each one's exact byte-sourcing). This project's dual-bitmap algorithm is only correct for `HIRES` (0x06) specifically. Fixed the trigger from `videoMode & 0x04` to `videoMode == 0x06`; the other three DECR values now correctly fall into the same "not implemented, render as standard mode" fallback as dual-file/hi-color, rather than silently running the wrong hi-res algorithm. The other three sub-modes are now precisely scoped for whoever implements them next (source their second byte from an attribute area instead of a bitmap area — exact FUSE line references in `SCLDVideo.h`), rather than vaguely "not confirmed."

**4. Standard-mode attribute-row addressing (`Video.cpp`, inline, untested) — the one that had been hiding in plain sight.** `attRowOffset = 0x1800 + row * 32` used the raw *pixel* row (0..191). Attributes are per 8×8 character cell and only change every 8 pixel rows; the correct shape (confirmed against FUSE's `display_attr_start[y] = 6144 + 32*(y/8)`) is `0x1800 + (row >> 3) * 32`. This would have read attribute data from a rapidly-drifting wrong offset almost immediately into any real screen — a standard-mode-breaking bug, not a hi-res edge case. Notably, the *bitmap*-row formula sitting right next to it in the same function was correct (independently hand-verified earlier against known Spectrum addressing facts, and now re-confirmed exactly against FUSE's `display_line_start[]` init loop) — the asymmetry is the lesson: hand-verification without an automated test is exactly how the attribute-row bug got through undetected. Both formulas are now `SCLDVideo::standardBitmapOffset()`/`standardAttributeOffset()` — pure, tested functions (5 checks each, including a same-character-row-shares-an-offset check that directly targets this bug class) — instead of unverified inline arithmetic in `Video.cpp`.

**Also checked and confirmed correct, no fix needed:** the `.DCK` chunk-type model (`peripherals/dck.c`'s `LIBSPECTRUM_DCK_PAGE_NULL/ROM/RAM_EMPTY/RAM` maps 1:1 to this project's `0x00/0x02/0x01/0x03`). **Noted, not fixed:** FUSE's `.DCK` loader supports HOME-bank RAM chunks (preloading HOME RAM content from a cartridge file); `DockLoader` still deliberately skips `BANK_HOME` entirely — a real, low-priority feature gap, not a bug, since this port's HOME RAM already comes from `RomLoader`/real hardware state rather than `.DCK` files.

**Forward note (no fix needed yet — keyboard port 0xFE isn't implemented in this slice):** the project owner flagged that Z80 8-bit port I/O (`OUT (n),A`/`IN A,(n)`) has a lesser-known 16-bit sibling — loading a full 16-bit value into BC and using `OUT (C),r`/`IN r,(C)` — and that real Spectrum/TS2068 keyboard-scan code leans on this for the row-select value. FUSE confirms the mechanism precisely: `peripherals/ula.c`'s port table decodes `0xFE` the same low-byte-only way as the port fix above (`{ 0x00ff, 0x00fe, ula_read, ula_write }`), but `ula_read()` then does `r &= keyboard_read( port >> 8 )` — pulling the row-select mask from the address's *upper* byte and handing it to `keyboard.c`'s `keyboard_read(libspectrum_byte porth)`. So port decode (low byte, mask-based, same shape as the 0xF4/0xFF fix) and the data a read actually needs (upper byte, for 0xFE specifically) are two separate concerns that happen to share one address. `Ports.cpp`'s `is2068`/`input()` switch now carries a comment citing this at the point where the keyboard port will eventually be added, so the upper-byte requirement isn't rediscovered the hard way later. This doesn't change the 0xF4/0xFF fix above — those two ports have no per-read data encoded in the upper byte, so `switch (address & 0xFF)` remains complete for them as-is.

All 6 host test suites re-verified after these fixes (31 SCLD / 15 DockLoader / 26 SCLDVideo / 14 RomLoader / 18 RealCartridge / 13 dck_format = 117 checks total, precisely counted, up from the low-90s before this pass — the new `standardBitmapOffset`/`standardAttributeOffset` checks and the corrected `expandHiresBytePair` checks account for most of the growth). `nopsram` firmware builds clean.

### Keyboard port 0xFE, read side (2026-07-10)

The forward note above is now acted on: `Ports::input()`'s `is2068` switch has a real `case 0x00FE`, not just a comment. It reuses `Ports::port[0..7]` (already populated every frame by `ESPectrum::processKeyboard()`, unconditionally of `Config::arch` — confirmed by reading `src/ESPectrum.cpp:2067`'s main-loop call site and the two unconditional `for (i<128) Ports::port[i]=0xFF` resets at `src/ESPectrum.cpp:879`/`:1013`, neither gated by arch) and copies the row-select-by-upper-address-byte fold from the existing Spectrum ULA branch a few lines below it in the same function. Two confirmed differences from that Spectrum path, not one: the idle default is the fixed byte `0x5F` rather than the Spectrum's Issue2/3-dependent `0xff`/`0xbf` (FUSE's `peripherals/ula.c`, `machine->timex` branch of `ula_write()`, hard-sets `ula_default_value = 0x5f` unconditionally for TS2068/TC2068 — meaning bits 5 and 7 always read 0, not float/undefined; independently corroborated by the TS2068 Technical Manual Sec. 2.1.13.2's port-FEH table, "D7/D5 not used"), and the port is fully low-byte decoded (matched via the same `address & 0xFF` already used for 0xF4/0xFF) rather than the Spectrum's single-bit `(address & 0x0001) == 0` partial decode — confirmed against FUSE's `machines_periph_timex()` (`machines/machines_periph.c:246-265`), which explicitly swaps in `PERIPH_TYPE_ULA_FULL_DECODE` (mask `0x00ff`) for Timex machines in place of the 48K's `PERIPH_TYPE_ULA` (mask `0x0001`), both routed to the *same* `ula_read`/`ula_write` functions — i.e. FUSE itself models the 48K and TS2068 keyboard read as one shared algorithm behind two different decode widths, exactly the shape this port now has. Also folded in bit 6 (tape EAR), matching FUSE's `ula_read()` doing this unconditionally for every machine type — a no-op today since nothing drives `Tape::tapeEarBit` off its `TAPEHIGH` (0) default for arch "2068" yet (cartridges are the loading path, not cassette).

The row-select-fold algorithm itself was not re-derived from scratch — it's the existing, already-shipped Spectrum ULA branch's loop, copied verbatim with only the idle default and decode width changed — so unlike the four FUSE-cross-reference bugs above (which were new inline arithmetic that had never been checked against anything), the risk profile here is "reuse a proven algorithm with two parameter changes," not "derive new addressing math." No new host test was added: `Ports.cpp` depends on `VIDEO`/`Tape`/`MemESP`/ESP-IDF machinery that's never been made to compile under the `ESP2068_HOST_TEST` host-test convention (unlike `SCLD.cpp`/`DockLoader.cpp`/`RomLoader.cpp`, which were deliberately kept free of that machinery for exactly this reason), and building that scaffolding just for a ~10-line verbatim-copied loop wasn't judged worth it. Verified instead by a clean `nopsram` build and a hand trace of the CAPS-SHIFT-only case (`B=$FE` selects half-row 0; `port[0]` with only bit 0 clear folds against idle `0x5F` to `0x5E` — bit 0 low/pressed, bits 1-4 high/unpressed, bits 5 and 7 forced low, bit 6 high/EAR-idle) against FUSE's confirmed `keyboard_data_table` mapping.

**Deliberately not done in this increment:** the write side of port 0xFE (border color in bits 0-2, tape MIC-out in bit 3, beeper in bit 4 — same layout as the Spectrum's write side per the Technical Manual) — border-color rendering isn't wired into `RenderFrame2068()` yet regardless (see slice 2's writeup above), and beeper/MIC output is a separate audio-path concern; both fall to `default: return;` (no-op) in `Ports::output()`'s `is2068` switch, same as before. AY sound ports (0xF5/0xF6) also remain unimplemented stubs.

### AY-3-8912 sound ports 0xF5/0xF6 (2026-07-10)

FUSE research first clarified a naming trap: FUSE actually has *two* Timex-shaped AY peripheral types, and only one applies here. `PERIPH_TYPE_AY_TIMEX` (`peripherals/sound/ay.c`'s `ay_ports_timex`, also ports 0xF5/0xF6) is the Spectrum SE's *second*, plain AY chip — no joystick logic, data port not even readable — and is registered only by `machines/spec_se.c`, never by TS2068/TC2068. The table that actually matters is `PERIPH_TYPE_AY_TIMEX_WITH_JOYSTICK` (`tc2068_ay_ports`, `machines/machines_periph.c:96-107`), registered `PERIPH_PRESENT_ALWAYS` (not optional) by `machines_periph_timex()` and pointed at Timex-specific read handlers defined in `machines/tc2068.c`, not `ay.c`. Getting this wrong would have meant modeling the TS2068's AY ports after the wrong FUSE machine entirely.

`src/Ports.cpp`'s `is2068` switches now have real `case 0x00F5`/`case 0x00F6` in both `input()` and `output()`. The write side is a straight port of the existing 128K pattern a few hundred lines below in the same file (`AySound::selectedRegister = data & 0x0f;` on 0xF5, `ESPectrum::AYGetSample(); AySound::setRegisterData(data);` on 0xF6) — confirmed against FUSE that the write handlers (`ay_registerport_write`/`ay_dataport_write`, `peripherals/sound/ay.c:189-211`) are shared, unmodified code between the 128K and both Timex AY tables, so there was nothing TS2068-specific to replicate on the write side at all.

The read side needed two real, TS2068-specific behaviors, both absent from the existing 128K read path (`src/Ports.cpp:435-440`, which only handles the register-select port at `(address & 0xC002) == 0xC000` and has no case for reading the *data* port back at all on plain 128K hardware):
1. **0xF6 (data port) is readable for any register**, register 14 included — confirmed against `tc2068_ay_dataport_read` (`machines/tc2068.c:58-82`), which for any register other than 14 just delegates to the generic `ay_registerport_read()`. This project's existing `AySound::getRegisterData()` (`src/AySound.cpp:516-545`) already implements that same generic per-register read (including the R7-direction-bit gating for R14/R15), so `case 0x00F6` is a single-line delegation to it.
2. **0xF5 (select port) always reads 0xff while register 14 is selected** — confirmed against `tc2068_ay_registerport_read` (`machines/tc2068.c:50-56`), which short-circuits register 14 to a hardcoded `0xff` *before* falling through to the same generic read the data port uses for every other register. This is the one piece of logic actually specific to the select port; implemented as `AySound::selectedRegister == 14 ? 0xff : AySound::getRegisterData()`.

**Joystick-via-register-14, deliberately not implemented.** FUSE's `tc2068_ay_dataport_read` also ANDs in inverted joystick state when R14 is selected, gated on address bits 8/9 of the specific `IN` instruction's upper byte (`if (port & 0x0100) ret &= ~joystick_timex_read(port, 0);` for the left/player-1 stick, bit 9 for player 2) — confirmed independently by the TS2068 Technical Manual's Sec. 2.4.4 wiring table and Sec. 4.3's sample `GETJOY` routine (`LD B,playerno; LD C,0F6H; IN A,(C)`, i.e. the player number selects which address bit gets driven, not an AY register). This project has no Timex-joystick config/input source anywhere yet (`ESPectrum`'s existing joystick handling is Kempston/Fuller/Sinclair-cursor, all completely separate machinery from the AY chip — confirmed by grep, nothing ties any existing joystick type to an AY register today). Rather than invent a new joystick-type/config surface as a side effect of "implement the sound ports," the fold was left out entirely — and left out is *correct*, not merely deferred-and-wrong: FUSE's own joystick state (`timex1_value`/`timex2_value`, `peripherals/joystick.c:65-66`) defaults to `0x00` (no buttons pressed) until something presses a button, and `ret &= ~0x00` is `ret &= 0xff`, a no-op. So the real HOME ROM's `STICK` routine reading port 0xF6 on this port today gets exactly the same answer a real TS2068 gives with nothing plugged into the joystick port — "centered, no button" — not an approximation of it.

**AY presence and audio-mixing prerequisite.** Unlike 48K/TK (`Config::AY48`, a user-facing on/off setting), FUSE's `machines_periph_timex()` registers the AY-with-joystick peripheral `PERIPH_PRESENT_ALWAYS` for TS2068/TC2068 — real hardware always has the chip. Tracing `ESPectrum::AY_emu`'s only two call sites (`src/ESPectrum.cpp`'s `setup()` and the arch-reload/reset path) found neither had a `Config::arch == "2068"` branch at all — meaning `AY_emu` stayed at its default `false`, and worse, the audio-mixing divisor constants (`samplesPerFrame`/`audioOverSampleDivider`/`audioAYDivider`/`audioSampleDivider`, all used as literal divisors in `BeeperGetSample()`/`AYGetSample()`) were *also* never set for this arch, a live divide-by-zero the instant a port write called `AYGetSample()`. Fixed by adding a `Config::arch == "2068"` branch to both copies of the chain (mirroring 128K/+2A/Pentagon's unconditional `AY_emu = true`), backed by new `ESP_AUDIO_*_2068` constants in `include/ESPectrum.h` derived from the already-established `TSTATES_PER_FRAME_2068` (58688, `cpuESP.h`): its prime factorization (`2^6 * 7 * 131`) happens to allow an exact `524 * 7 * 16 = 58688` split (`samplesPerFrame=524`, `audioSampleDivider=7`, `audioOverSampleDivider=16`, `audioAYDivider=112`, matching 48K/Pentagon's own `16*7=112` divisor pair exactly), which lands the final output rate on a clean `3528000 / 112 = 31500`Hz — the same "perfect division" shape the header's own 48K comment calls out, not a coincidence but the reason this particular factor split was chosen over other valid divisor combinations. `524 < ESP_AUDIO_SAMPLES_PENTAGON` (640, the shared buffer-sizing constant), so no buffer-overrun risk. 125%/150%-speed `Audio_freq[]` variants were filled in identically to every other arch's branch, scaled from the same 31500Hz base (39375/47250), for consistency with the CPU-speed feature that already exists generically.

No new host test, same reasoning as the keyboard port increment above: `Ports.cpp` isn't part of the `ESP2068_HOST_TEST` set, and both the write-side and read-side logic here reuse already-shipped `AySound`/`ESPectrum` methods rather than introducing new derived arithmetic (the one piece of genuinely new logic, the R14-on-select-port special case, is a single ternary, hand-verified against FUSE's `tc2068_ay_registerport_read` line-by-line). Verified by a clean `nopsram` build (full recompile since `include/ESPectrum.h` changed, `SUCCESS`, no new warnings beyond PlatformIO's own pre-existing `pkg_resources` deprecation notice).

### Printer port 0xFB (2026-07-10)

Unlike keyboard/AY, this port had nothing to reuse: an exhaustive grep across `src/`/`include/` for anything printer-related (before this increment) found exactly one hit, the placeholder comment `// Other TS2068 ports (printer $FB, etc.): not this slice's scope.` — no ZX Printer, no TS2040, no generic printer class, no SD-file-write helper (`FileUtils.h` has no `openForWrite()`-style wrapper; every existing write path in this codebase — `CaptureBMP.cpp`, `Snapshot.cpp`, `Tape.cpp` — calls raw `fopen(path, "wb")` directly). This was greenfield work, closer in shape to slice 2's `SCLDVideo` (new pure host-testable math class) than to the keyboard/AY ports (wire an existing engine into a new address).

**FUSE has a naming trap here too, structurally identical to the AY one.** `PERIPH_TYPE_ZXPRINTER` (`peripherals/printer.c`'s `printer_zxp_ports`, mask `0x0004`/value `0x0000` — the 48K's classic partial decode, "any address with bit 2 clear") is the plain accessory-port printer every Spectrum machine can have. `PERIPH_TYPE_ZXPRINTER_FULL_DECODE` (`printer_zxp_ports_full_decode`, mask `0x00ff`/value `0x00fb`) is the *same* read/write handler functions, registered separately by `machines_periph_timex()` for TC2068/TS2068 with a tightened decode — full low-byte match, same "why" as every other TS2068 port in this project (`address & 0xFF`, not a partial-bit test). Confirmed against the TS2068 Technical Manual (Sec. 2.1.13.3): `"the Printer responds to any I/O Read/Write with Address Bit 7=1 and Address Bit 2=0... Ports 0FA through 0F8 and Ports 0F3 through 0F0... will interface to the Printer"` on real 48K-style hardware, but the TS2068's own on-board decode is the tightened one — 0xFB and only 0xFB. **A second, separate trap**: FUSE's shared `ay_dataport_write()` has `if(current==14) printer_serial_write(b);`, which looked at first glance like a *second*, AY-register-14-based Timex printer path (an "RS232/Alphacom" hypothesis). Both FUSE's own man page (`PRINTER EMULATION` section: *"16, 48, TC2048, TC2068, TS2068 → ZX Printer; 128/+2/Pentagon → Serial printer"*) and the Technical Manual (Sec. 2.1.6/2.1.6.1/2.1.7, three separate passages, all unambiguous: register 14/port F6 is joystick I/O, full stop) rule this out — the AY-14 printer hook is live, reachable code in FUSE, but it's an artifact of one write function being shared across every AY peripheral table, not a modeled TS2068 hardware feature. Confirms the AY-port increment's earlier decision (register 14 = joystick, not printer) was already correct.

**The protocol itself is not a byte-in/byte-out register at all.** Both FUSE's `printer_zxp_read`/`printer_zxp_write` (`peripherals/printer.c:437-577`) and the Technical Manual's Sec. 2.1.13.3 bit tables describe a synchronous, software-timed stylus/motor simulation: a write's bits are *sampled* (bit7 = stylus down/up = pixel black/none, bit1 = paper-feed speed 0=fast/1=slow, bit2 = motor stop), and the "current print-head x-position" is derived purely from elapsed CPU cycles since the motor last started or crossed a line boundary — not from any byte payload. A read's status byte reflects that same derived position: bit7 set means busy (either genuinely mid-stylus-contact, or in the `-10 < x < 0` pre-paper lead-in window), bit0 set means "the head has moved to a new pixel position since the last write" (a ready-for-next-sample ack real ROM code paces itself against). Getting this arithmetic right is what makes HOME ROM's `PRSCAN`/`K_DUMP` (Technical Manual Sec. 4.1.3, confirmed against real ROM disassembly at `$0A08`-`$0A82`: `OUT ($FB),A` for pixel/motor bits, `IN A,($FB)` for status) poll correctly instead of hanging or racing.

`include/SCLDPrinter.h` + `src/SCLDPrinter.cpp` are an original reimplementation of this *behavior* (not a port of FUSE's code — different license, GPL-2 vs this project's GPL-3, and a deliberately narrower scope, see below), kept fully framework-agnostic like `SCLDVideo` (no ESP-IDF, no `CPU.h`/`Video.h` dependency) so it's host-testable. The four physical-geometry constants (`CYCLES_PER_PIXEL_FAST`/`_SLOW` = 220/440, `LINE_PITCH` = 384, `LINE_ADVANCE_THRESHOLD` = 320, `HEAD_LEAD_IN` = 64) were cross-checked against FUSE's equivalent literals (`440/zxpspeed`, the `x-=384`/`x>320` line-wrap arithmetic, the `-64` origin offset) rather than re-derived independently — this is real device geometry, not something to guess at twice. The "limit height of blank paper" staleness clamp (FUSE: `if(frame>400) frame=400`) is reproduced as a single `MAX_ELAPSED_CYCLES = 400 * 58688` (TS2068's `TSTATES_PER_FRAME_2068`, duplicated as a literal rather than included, matching `SCLDVideo.h`'s zero-dependency convention) since this class tracks one monotonic cycle count rather than FUSE's separate (frame, in-frame-cycle) pair — a real simplification enabled by `CPU::global_tstates` already existing in this codebase as exactly that whole-session counter (confirmed via `Tape.cpp`, which already uses `CPU::global_tstates + CPU::tstates` for its own timing — the same idiom `Ports.cpp`'s new `case 0x00FB` now reuses rather than inventing a second one).

**Deliberately, explicitly not implemented: persisting any printed output.** FUSE's `printer_zxp_write()` does two things per write — advance the timing state machine (what this port replicates), and fill a `zxpline[256]` pixel buffer that eventually gets written to a PBM/text file via `printer_zxp_output_line()` (what this port does not replicate at all). The timing half is what real software's correctness depends on — a print routine that never got a valid "ready" bit back would hang, which is the failure mode worth preventing first. The pixel-persistence half is a distinct feature (accumulate a page, decide a file format and an SD path/naming convention, flush on motor-stop) with no bearing on whether guest software runs correctly, closer in shape to `CaptureBMP.cpp`'s auto-numbered-file convention than to anything already in this class — deferred as genuinely separate future work, not silently dropped. This also settles the "is a printer attached" question in the other direction from FUSE's own default: FUSE ships with printer *capture* off by default (returns `0xff`, "not configured," until the user opts in), but `SCLDPrinter` always behaves as an attached, responding printer (idle-at-boot, correct status protocol) — ESP2068 has no config surface for "user chose not to attach one," and the task was to make the port work, matching this project's established pattern from the AY-joystick decision (behave as present-and-idle, not as absent).

`test/host/scldprinter_test.cpp` (15/15 checks) covers: idle-forever, motor start with the head still in lead-in, the `-10<x<0` busy window with the ready-bit correctly still clear, `x==0` paper-reached with the ready-bit correctly set, stylus-down forcing busy regardless of x, motor stop returning to idle immediately and staying there, `reset()` forcing idle mid-print, stop-while-already-stopped being a harmless no-op (not a spurious start), and two boundary scenarios (a multi-line-crossing time jump, a mid-print speed change) asserted only as "still a valid status byte, doesn't hang" rather than an exact expected value — hand-deriving the precise byte for either scenario requires exactly replicating the write()/read() while-loop's iteration count and one-shot pending-speed application, and the risk of a wrong hand-trace silently validating a broken implementation (or rejecting a correct one) was judged higher than the value of a tighter assertion there; flagged here rather than left implicit. All 7 host test suites re-verified after this increment: 31 SCLD / 15 DockLoader / 26 SCLDVideo / 14 RomLoader / 18 RealCartridge / 13 dck_format / 15 SCLDPrinter = 132 checks total. `nopsram` firmware builds clean (`SUCCESS`, confirmed `SCLDPrinter.o` actually compiled and linked, no new warnings).

### Port 0xFE write side: border/beeper/tape-out (2026-07-10)

With four ports done (0xF4, 0xFF, 0xFE-read, 0xF5/0xF6, 0xFB), the remaining TS2068 port map split into two very different pieces: finishing port 0xFE's write side (border color, beeper, tape-MIC-out — mainstream hardware every piece of software touches), versus a six-port expansion-bank cluster (`$A0` Universal Deselect, `$40` HS, `$80` BNA, `$C0` CMD, `$FC` BDATPT, `$FD` BCMDPT — confirmed via the reference library's `ts2068_memory_map.md` I/O port table and `ts2068_video_and_cartridges.md`'s note on "up to 11 expansion bank entries" following the LROS header entry, dispatched via `GOTO_BANK`/`CALL_BANK`). The latter is a distinct, AROS-multi-bank-cartridge-specific bank-switching subsystem — separate from the DOCK/EXROM chunk paging `$F4`/`SCLD.cpp` already handles — and a much larger, more speculative research+implementation effort than a single port. Asked the project owner which to build; **the recommendation (port 0xFE write side) was confirmed** rather than guessing at scope for the expansion-bank cluster, which stays an open, explicitly-tracked gap (see CLAUDE.md's "not yet done" list) rather than something silently skipped.

The write itself needed no new state machine, unlike the printer port — it's the same bit layout as the Spectrum ULA's own port 0xFE write (Technical Manual Sec. 2.1.13.2: bits 2-0 border color, bit 3 tape/MIC out, bit 4 beeper, bits 5-7 unused), and the underlying beeper-mixing formula (`speaker_values[((data>>2)&0x04)|(Tape::tapeEarBit<<1)|((data>>3)&0x01)]`, `ESPectrum::BeeperGetSample()`) and `VIDEO::borderColor`/`VIDEO::brd` state fields are already arch-agnostic code the Spectrum ULA branch a few hundred lines below already exercises — copied verbatim into a new `case 0x00FE` in `Ports::output()`'s `is2068` switch rather than re-derived. The one prerequisite this increment depended on — correct `samplesPerFrame`/`audioOverSampleDivider`/`audioAYDivider`/`audioSampleDivider` constants for `Config::arch == "2068"` — was already fixed during the AY-port increment, so this produces genuinely audible beeper output on real hardware, not a stub that compiles but stays silent.

**What was deliberately left out, and why it's provably safe to leave out:** the Spectrum ULA branch also calls `VIDEO::DrawBorder()` and Pentagon/TK-specific contention-timing `VIDEO::Draw()` helpers inline, every time the border color changes — part of the cycle-accurate, self-retargeting scanline renderer (`Draw`/`Draw_Opcode` function pointers reassigned mid-frame as `tstates` crosses scanline boundaries) that TS2068 never uses at all (`RenderFrame2068()` batches the whole frame once per frame instead, per slice 2's writeup above; `Draw`/`Draw_Opcode` stay pointed at `&Blank`/`&Blank_Opcode` the entire TS2068 session). Calling `DrawBorder()` from this new case would mean invoking a function tuned for the Spectrum's border+active-area geometry against TS2068's fixed 512-column framebuffer, with no verification either way that's safe — so it wasn't called. `VIDEO::brdChange` *is* still set on a real border-color change, and this was verified safe rather than merely assumed: `grep`ing every consumer of `VIDEO::brdChange` found exactly two, `VIDEO::EndFrame()` (`src/Video.cpp:1032`) and the `MainScreen_Blank`-family `Draw_Opcode` functions (`src/Video.cpp:494/524/557/594`) — the former is never called for `is2068` (`RenderFrame2068()` runs instead, at the exact call site CLAUDE.md's slice-2 writeup documents), the latter are never reachable either since `Draw`/`Draw_Opcode` never retarget away from `Blank` for `is2068`. So `brdChange` is a genuinely dead flag on this code path today, not a risk — kept updated purely so a future border-rendering slice for TS2068 starts from correct state instead of needing to backfill it.

No new host test: this reuses already-shipped, already-arch-agnostic `VIDEO`/`ESPectrum`/`Tape` state and formulas verbatim (same reasoning as the keyboard/AY increments — `Ports.cpp` isn't part of the `ESP2068_HOST_TEST` set, and there's no new derived arithmetic here to test in isolation). Verified by a clean `nopsram` build (`SUCCESS`, no new warnings).

### Expansion-bank port cluster (`$A0`/`$40`/`$80`/`$C0`/`$FC`/`$FD`) — scoped, and closed with no code change (2026-07-10)

Three independent research passes (the local FUSE 1.9.0 checkout; the TS2068 reference library's Technical Manual + ROM disassemblies; this project's own existing code/docs) converged on the same conclusion, strongly enough to close this out as a deliberate non-implementation rather than leave it as an open "not yet done" item.

**The hardware never shipped.** This cluster would have driven Timex's "Bus Expansion Unit" (BEU) — a general-purpose bank-switching bus for hypothetical third-party accessories, addressed by bank numbers 1–253 alongside the built-in HOME(255)/DOCK(0)/EXT(254) banks, dispatched through the HOME ROM's `GOTO_BANK`/`CALL_BANK`/`BANK_ENABLE` services (System Configuration Table at `$5EEA`, `MAXBNK` system variable). The TS2068 Technical Manual's own Port Map (Table 2.1.13-1) labels ports `$FC`/`$FD` *"Reserved... Bank switching - not implemented"* outright; `$A0`/`$40`/`$80`/`$C0` don't appear in that table at all — the `ts2068_memory_map.md` names for them ("Universal Deselect Byte," "HS," "BNA," "CMD") are this project's own reference library's inferred glosses, not terminology found anywhere in the primary Timex source. The reference library's own errata doc says it plainly: *"These services were designed for the Bus Expansion Unit (BEU) which was never produced."* A deeper look at the actual EXROM disassembly turned up something the simplified port-map doc gets wrong in a way that would have mattered had this been built: the ROM never issues real Z80 `IN`/`OUT` to $40/$80/$A0/$C0 at all — `WRITE_BS_REG`/`READ_BS_REG` instead treat them as the *upper byte of a memory address* (`$4000`/`$8000`/`$A000`/`$C000`), written via a nibble-split protocol that reprograms the AY sound chip's I/O-port-A direction as a side channel. Implementing this from the reference library's own port-map framing alone would have trapped the wrong bus cycle type entirely.

**No reference implementation exists anywhere to check against.** FUSE — a mature, ~25-year, actively-maintained, GPL codebase that has emulated the TS2068 (via `peripherals/scld.c`, `machines/ts2068.c`/`tc2068.c`) for essentially its whole life, and which this project has repeatedly used to catch real bugs (see the "Cross-referencing FUSE" section above) — registers zero ports for any of this cluster, and neither its source, its docs, its man page, nor its ChangeLog mention "expansion bank," "BNA," "BEU," "AROS," or "Command Cartridge" anywhere. Confirmed mechanically too: FUSE's generic unhandled-port behavior (`periph.c`'s `readport_internal()`/`writeport_internal()`, `machine->unattached_port` hardcoded to `0xff` for Timex machines) is exactly what a real, accessory-less TS2068 gets on these ports today — there was never a gap in FUSE to learn from, because there was never a need.

**No real or modern software uses it.** Every genuine cartridge/accessory this project's own reference library documents in depth was checked: the real, shipped Zebra OS-64 cartridge (full manual text searched) — no hits. The real, shipped Timex FDD3000 floppy drive — uses its own bespoke protocol on an unrelated port (`$EF`), not this mechanism. The GUS enhanced ROM, eToolkit — no hits. Most tellingly, a currently-active (2023) homebrew SD-storage project for this exact machine, TS-PICO, invented its own protocol on entirely different ports rather than reviving the BEU scheme — consistent with no hardware designer, then or since, ever having a real BEU card to build against.

**The current default behavior is already correct, not a stopgap.** `Ports.cpp`'s existing `default:` fallthrough for unhandled TS2068 ports — `0xFF` on read, silently discarded on write, the same behavior every other not-yet-implemented TS2068 port already has — is indistinguishable from a real TS2068 with no BEU plugged in, which describes every unit Timex ever sold. The ROM's own bank-detection self-test (`$0BD1`, "find all used banks") pokes a probe value through the same nibble-protocol path and backs out gracefully to `MAXBNK=0` when nothing answers — exactly what happens today. No code change makes this *more* correct; the honest, evidence-backed statement is that it's already right.

**What would still be worth doing, and isn't blocked by any of this:** `GOTO_BANK`/`CALL_BANK` are not BEU-exclusive — confirmed via their pervasive use throughout both ROMs for ordinary HOME/DOCK/EXT bank switches (bank values 255/0/254), which `BANK_ENABLE` routes through the already-implemented `$F4`/`$FF` ports, not this cluster. So if/when this project ever runs real HOME ROM code (blocked on other, already-tracked gaps — no OSD ROM picker, no real hardware/monitor to verify against), nothing about that path needs anything built here. Only bank numbers 1–253 would ever reach the unimplemented ports, and no real software has a reason to request one.

Decision: closed with **no code change**. `CLAUDE.md`'s "not yet done" framing is retired in favor of a short "scoped and deliberately not implemented" note pointing back here, since listing this as a lingering TODO would misrepresent a researched, deliberate decision as unfinished work.

### Real bug found and fixed: EXROM was wrongly modeled as mirrored, not chunk-indexed (2026-07-11)

Found while scoping `TS2068-VIRTUAL-DISK-PLAN.md`'s "extended EXROM size and layout" open item — extending EXROM only makes sense once its addressing model is actually correct, so this had to be resolved first.

**The original assumption, and why it was reasonable at the time.** `SCLD.h`/`SCLD.cpp` (slice 1) modeled EXROM as a single fixed 8K image, chip-select-mirrored across every chunk that selects it — `exromPage(int chunk)` ignored its own `chunk` parameter entirely, unconditionally returning the same pointer. The reasoning, quoted from the code at the time: *"On real hardware its own address decode is 13 lines wide (chip-select gated by the SCLD, not by which Z80 chunk selected it)... there's no larger EXROM address space behind it for a second bit to index into."* This was a genuine, considered judgment call — the only real EXROM dump available to test against at the time was the stock 8K `2068Exrom.BIN`, and mirrored-vs-chunk-indexed addressing is completely indistinguishable from an 8K-only sample; there was no way to tell which was true from the evidence on hand.

**The contradicting evidence.** The project owner's own real TS-Pico EXROM replacement chip, `gus-exrom.rom` (in the local reference library, alongside its `gus-rom-analysis.md` writeup already used for the virtual-disk design), is genuinely **16384 bytes — exactly double** the stock 8192. And that analysis doc's own "Power-Up Chunk Select Changed at $0049" finding says the boot patch changes the Horizontal Select Register write from `$01` (chunk 0 only) to `$03` (chunks 0 *and* 1 together) specifically to make *"the EXROM's extended code area... accessible during initialization."* Activating a second chunk only makes sense if it shows *different* content — an identical mirror would be redundant, you'd just leave chunk 0 alone. Chunk 0 (`$0000-$1FFF`) + chunk 1 (`$2000-$3FFF`) = exactly 16K, matching the real file size precisely. Real EXROM chip-select is per-chunk, exactly like `DOCK`'s model — which this project got right from the start, since `.DCK` cartridges' own chunk-typed structure never let a mirrored assumption creep in there.

**The fix**, mirroring `dockPage()`/`loadDockChunk()`'s shape exactly: `exromImage` (single pointer) became `exromChunk[8]` (one pointer per chunk, `nullptr` until loaded). `exromPage(int chunk)` now actually uses its parameter. Two new per-chunk primitives, `SCLD::loadExromChunk(int chunk, uint8_t*)`/`unloadExromChunk(int chunk)`, mirror `loadDockChunk()`/`unloadDockChunk()` (minus the writable flag — EXROM is always ROM). A convenience wrapper, `loadExromImage(uint8_t* data, int numChunks)`, slices one contiguous whole-file buffer into up to 8 chunk pointers via plain pointer arithmetic, for `RomLoader`'s simpler "one file, one image" case.

**This also fixed a second, latent instance of the same bug.** `DockLoader.cpp`'s `.DCK` loader already iterates its `BANK_EXROM` entries per-chunk (`.DCK`'s own format is chunk-structured for EXROM too, same as DOCK) — but under the old single-pointer model, each `SCLD::loadExromImage(buf)` call for a second or later EXROM chunk silently *overwrote* the previous one rather than adding to it. A `.DCK` file with a multi-chunk EXROM bank would have silently lost all but its last chunk. Fixed by calling the new `loadExromChunk(pc.index, buf)` per chunk instead, exactly mirroring the adjacent `BANK_DOCK` branch's own `loadDockChunk(pc.index, ...)` call one line above it.

**`RomLoader::loadExromFromBuffer()`/`loadExromFromFile()`** now accept any whole multiple of 8K from 8K to 64K (previously: exactly 8192 bytes, nothing else), computing `numChunks` and delegating to `SCLD::loadExromImage()`.

**Verification, not just re-derivation of the same claim twice:** `test/host/scld_test.cpp` gained checks that two loaded EXROM chunks resolve to genuinely different pointers (not a repeat of the same one), plus coverage of the `loadExromImage()` convenience wrapper's slicing and its "chunk beyond `numChunks` falls back to the empty socket" boundary (35/35 checks, +4). `test/host/romloader_test.cpp` gained size-validation checks for the new range, a synthetic 16K (2-chunk) round-trip confirming chunk 0 ≠ chunk 1, and — the most direct confirmation available — a new optional check that loads the *real* `gus-exrom.rom` and empirically verifies its chunk 0 and chunk 1 contain genuinely different bytes, not a synthetic stand-in for the claim (21/21, +7). `test/host/dockloader_test.cpp` and `test/host/real_cartridge_test.cpp` re-verified unchanged (neither exercises EXROM directly, but both touch `DockLoader.cpp`, which this fix modified). All 7 host suites: 143 checks total, up from 132. `nopsram` build clean, no new warnings.

## Phase 4 — TS2068 Virtual Disk

[TS2068-VIRTUAL-DISK-PLAN.md](TS2068-VIRTUAL-DISK-PLAN.md) (v1.0, all open items resolved) is this phase's technical contract — the five ROM patch points, the `$0E`/`$0F` wire protocol, the command syntax, and the reused-vs-new inventory all live there. Unlike Phase 2's four slices, this is a single new feature with no upstream equivalent at all — patching real ROM dead-ends and underspecified entry points to talk to the SD card as a virtual disk, inspired by the FDD3000's command set but not emulating its hardware (see the design doc's "Why not real FDD3000 emulation").

**Prerequisite housekeeping — resolved (2026-07-11), with a citable source, not just an assumption.** A web search for public corroboration of "Timex released the TS2068 ROMs to the public domain" (World of Spectrum's permits page, the Spectrum FAQ, the `timexcomputerworld.com`/`loadzx.com` archives, Timex Computer Corporation's 1984 corporate history) found nothing either confirming or denying it — worth recording that the search came up empty rather than silently skipping it. What actually settled this: the project owner produced a real, on-file, attributable statement from **Louis M. Galie, Senior Vice President of Timex Group USA Inc.** (the modern corporate successor holding whatever rights descend from Timex Sinclair/Timex Computer Corporation), made when TS2068 archivist Bruno Florindo asked permission to scan and preserve Timex USA material: *"if it came from Timex or Timex Computer company and it is not otherwise marked as belonging to someone else, you can disseminate it freely."* Full quote and citation now live in `test/dock/README.md`'s "Provenance policy" section (the authoritative copy) and `TS2068-VIRTUAL-DISK-PLAN.md`'s "Provenance note"; `CLAUDE.md` updated to match. Scoped precisely, per Galie's own carve-out ("we can not... extend this permission to... materials for which we do not have ownership"): covers Timex's own stock `2068Home.BIN`/`2068Exrom.BIN` and this project's patched derivatives of them — does **not** cover the reference library's third-party content (TS-Pico's `gus-home.rom`/`gus-exrom.rom`, `eToolkit.ROM`, `Zebra OS-64.BIN`), which stays under the original conservative policy. Both open decisions blocking Pieces B/C are now cleared.

**A genuinely new kind of work for this project — flagged explicitly rather than folded silently into "implementation."** Every ROM-adjacent increment so far (`RomLoader`, `DockLoader`, the FUSE cross-reference pass, all five patch points' own hex-verification) has been either C++ emulation code or read-only research against existing dumps — hex-tracing real bytes, never authoring new ones. Pieces B and C mean producing new, correct Z80 machine code and splicing it into a copy of the real ROMs — a different skill and a different risk profile than anything in Phase 0–3.

**Z80 assembly tooling — resolved (2026-07-11): `sjasmplus`, verified rather than assumed.** Both hand-assembly and two candidate assemblers were on the table. Hand-assembly was dropped first — fine for a single 3-byte `JP` (already done, by hand, for all five patch points' *identification*, since that only required decoding existing bytes, not encoding new ones), not something to accept for potentially hundreds of bytes of new dispatch/parsing/protocol code across two new EXROM routines and two HOME ROM routines, where a slipped relative-jump displacement or byte-order mistake would be silent and hard to catch without rebuilding this project's entire hex-verification discipline by hand for every new byte. Between the two real assemblers found already present on this machine — `sjasmplus` v1.22.0 at `/Users/david/bin/sjasmplus` (a manually-placed arm64 binary, not from Homebrew — no `sjasmplus` formula/tap exists; presumably fetched from the project's own GitHub releases, `z00m128/sjasmplus`) and the classic GNU `z80asm` v1.8 (`/opt/homebrew/bin/z80asm`, GPL-3.0-or-later, a general assembler+linker toolchain oriented at building whole programs from object files) — `sjasmplus` was chosen and proved, not just picked by reputation, with two throwaway assembly tests in `scratchpad/z80test/`:

1. A tiny synthetic "ROM" (32 recognizable bytes), patched with `INCBIN "fake_rom.bin", 0, 10` / a 3-byte `JP intercept` / `INCBIN "fake_rom.bin", 13, 19`, with `intercept:` new code appended after the original image's end. `--raw` output confirmed byte-exact: original bytes 0–9 and 13–31 untouched, `C3 20 00` (`JP $0020`) at the redirect point, `3E 42 C9` (`LD A,$42` / `RET`) at the new label's real computed address.
2. The real shape this project needs: an 8K synthetic EXROM "chunk 0" with a 3-byte redirect at a `$0068`-style offset, and new "chunk 1" code placed via a hard `ORG $2000` (the exact Z80 address chunk 1 maps to once SCLD's `$F4` selects it — not "wherever the assembler happens to land after the `INCBIN`"). Output confirmed: `C3 00 20` (`JP $2000`) at the redirect, `3E 99 CD 99 0F C9` (`LD A,$99` / `CALL $0F99` / `RET`, including a call to a fixed absolute address matching the real `PGPSTR` trampoline location) at `$2000` exactly, and real content byte-exact on both sides of the 3-byte patch (`$0067`/`$006B` both matched their expected untouched values).

This means **one tool does the entire job in one pass** — `INCBIN` splices real ROM content directly into the same source file as the new code, `ORG` places new code at exact required addresses (chunk boundaries, or wherever HOME ROM dead space turns out to be), and `--raw` emits a finished derivative binary. No separate patch-application script is needed after all — the "closer in shape to `dck_format.py`'s binary manipulation" framing this decision originally expected turned out not to apply; `sjasmplus` absorbs that step. The `.asm` source is what gets committed and reviewed; the derivative `.bin` is a build artifact (regenerate via `sjasmplus --raw=<out> <source>.asm`, matching `RomLoader`'s existing "plain binary in, plain binary out" convention with no new intermediate format). `z80asm` was not disproven, just not needed once `sjasmplus` was shown to cover the whole workflow single-handedly — no reason to add a second assembler dependency.

### Piece A — ESP32-side infrastructure

New `$0E` (command)/`$0F` (status) port handlers in `Ports.cpp`'s `is2068` switches, following the same full-low-byte-decode convention every other TS2068 port already uses. A new C++ class tracks mount state (which SD file is mounted, current block/position) the same shape `Tape`'s `tapeFileType`/`tapeCurBlock` already do — per the design doc's "Where state lives," almost none of this needs new TS2068 RAM, since the ESP32 host is a persistent process across Z80 instructions. Command handlers for `MOUNT`/`WRITE_HEADER`/`WRITE_DATA`/`READ_HEADER`/`READ_DATA`/`CAT_SD_LINE`/`CAT_CONTAINER_LINE`/`UNMOUNT` read parameters directly out of Z80 registers (`Z80::getRegIX()`/`getRegDE()`, the same accessors `Tape::FlashLoad()`/`Save()` already use) and emulated memory, no byte-serial framing needed since both sides run in the same process. Three sub-pieces of real, new logic, all called out in the design doc's "Reused vs. new": the SD-directory `CAT` listing (fast path only — reads `FileUtils`'s cached index file, does not trigger a live `DirToFile` rebuild), the mounted-container `CAT` listing plus a new name-based block-lookup layer built on top of `Tape.cpp`/`Tape_TZX.cpp`'s existing (reusable, pure) block-parsing/indexing engine, and one-shot `SAVE "D:..."` built on `Tape::Save()`'s existing `.tap`-block-writing logic with its one `MemESP::readbyte()` call site substituted for a TS2068-appropriate read (design doc flags this as TBD which layer — `SCLD` memory access or the patch intercept's own IX-relative read — actually needs the substitution; resolve during implementation, not here). `.tzx` writing is explicitly out of scope, matching the design doc's "no `.tzx` output" decision — `SAVE "D:x.tzx"` should error cleanly.

### Piece B — EXROM patches

Three patch points, all hex-verified in the design doc against `2068Exrom.BIN`: `W_TAPE` (`$0068`, 3-byte `JP` over `LD HL,$00E5`), `R_TAPE` (`$00FC`, 3-byte `JP` over three clobbered 1-byte instructions the intercept must replicate before falling through to real tape I/O), and the `PGPSTR`-call redirect (`$0215`, replaces `CALL $0F99` — the intercept still calls the real `$0F99` first, then inspects `(DE)` for a `"D:"` prefix and captures pointer+length into new system variables mirroring TS-Pico's `TPNAME`/`TPNLEN` pattern). Each of the three intercepts does its own `IN A,($F4)` / `OR` chunk-1's bit / `OUT ($F4),A` / restore around any call into the new, chunk-1-resident code — the design doc's resolved answer to EXROM reachability, chosen specifically so no HOME ROM boot patch is needed and the "EXROM-only for the tape commands" scope holds. The new chunk-1 code itself (a 16K total EXROM, one extra chunk beyond stock, matching TS-Pico's own proven real-world size for a comparably-scoped feature) implements the `"D:"` prefix parse and the command dispatch that drives Piece A's port protocol. Depends on Piece A's port numbers/command set being frozen, since this code is what issues the `OUT ($0E),A` calls.

### Piece C — HOME ROM patches

Two patch points, both hex-verified against `2068Home.BIN`: `CAT` (`$25C8`, a 4-byte dead-end redirect — needs no EXROM/`SLVM` involvement at all, just "talk to the new port, print the response" through HOME ROM's own native character-output routine) and `CLOSE #4` (`$139F`'s `CALL $140F` redirected to an intercept that replays the real `$140F`, checks `(STRMN)` for `4`, and either leaves `BC` untouched for every other stream or forces `BC=0` after unmounting — see the design doc's "CAT and CLOSE" for the full hex trace of why `STRMN` and not `$140F`'s resolved-channel return value is the right thing to check). Both patches are small, self-contained redirects of already-dead or already-safe-to-repurpose code, the same size/risk class as Piece B's three — not a `DOSAVE`-style rewrite. Independent of Piece B (different ROM file), but shares Piece A's port dependency for the same reason.

### Piece D — Integration test

Real patched EXROM/HOME ROM binaries plus the live ESP32-side handlers, exercised together for the first time — `LOAD "D:file.tap"` mounting a container, bare `LOAD "progname"` addressing a block within it, `SAVE "D:file.ext"` one-shot write, `VERIFY`/`MERGE "D:..."`, `CAT` in both its SD-directory and mounted-container modes, `CLOSE #4` unmounting. Given this project has no host-testable way to run real Z80 machine code against patched ROM content today (`CPU.cpp`'s Z80 core is ESP-IDF-entangled, unlike `SCLD`/`DockLoader`/`RomLoader`), the realistic exit bar for this piece is a `nopsram` build plus whatever on-device verification is possible when hardware is available — same honest gap Phase 0/3 already carry for the base port, not a new one invented here. A lighter-weight interim check worth doing regardless of hardware access: a script asserting the patched ROM bytes at each of the five confirmed offsets match the intended new opcodes, the same hex-level rigor the design doc's own verification already used, catching "the patch didn't land where intended" independently of whether a Z80 core ever executes it.

### Exit criteria

- **Piece A:** command handlers for all eight `$0E` operations exist and are host-testable via synthetic port-I/O sequences (no patched ROM required to exercise them); SD-directory and mounted-container `CAT` listings both produce correct output against real files; name-based block lookup finds a real block in a real `.tap`/`.tzx` fixture; one-shot `SAVE` produces a byte-correct `.tap` block.
- **Piece B:** all three EXROM patch points land as clean redirects (verified at the hex level against the patched binary, not just "the assembler didn't error"); the chunk-1 toggle-and-restore leaves `$F4` exactly as it found it on every exit path, including error paths; non-`"D:"`-prefixed filenames fall through to real, unmodified tape I/O byte-for-byte.
- **Piece C:** `CAT`'s two modes both reachable from the patched entry point; `CLOSE #4` unmounts and leaves every other stream number's behavior byte-for-byte identical to stock.
- **Piece D:** end-to-end command flow verified in software to the extent possible without a board (same caveat as every other phase); real hardware boot-and-use remains open, tracked the same way Phase 0/3's hardware gaps already are.

## Open decisions

These are called out in the plan doc too; repeating here because they affect sequencing, not just architecture:

- **TS2068 (60Hz/NTSC) vs TC2068 (50Hz/PAL):** build 60Hz first (it's the target). PAL is a timing-profile variant on the same `Reset()` switch slice 1 already owns — cheap to add after, not worth blocking v1 on.
- **Extended colour mode (010):** recommend deferring past v1. Standard + hi-res cover most native software; this shrinks slice 2's v1 scope without losing much real-world compatibility.
- **Board:** nopsram (TTGo VGA32-class) is the primary target and what the memory budget in the plan doc assumes. Keep one PSRAM board on the bench for double-buffering experiments only — don't let it become a dependency for v1.
- **Repo hosting (new, not in v0.1):** local git history with an `upstream` remote is enough to start (Phase 0). Whether/when to push this to a GitHub fork under your account is a separate call — flag it when you're ready to collaborate with the rest of the team or want CI, since that's the point a fork or a fresh repo with `EremusOne/ESPectrum` credited becomes worth setting up.

## Risk register

- **VGA timing for the new 512-column mode is hand-derived, not copy-paste.** Existing `vidmodes[]` rows top out at 360 active columns and each hand-tunes six APLL clock-divider constants plus porch/sync widths. Budget real monitor time for slice 2, not just a spreadsheet calculation.
- ~~**No existing per-chunk read-only mechanism.**~~ **Resolved by slice 1.** v0.1 of the plan doc assumed ESPectrum's ROM write-protect "generalizes"; verification found it's a single hardcoded `page == 0` check, not a flag array. `SCLD::memChunkReadOnly[8]` is the new one, built fresh as expected, and is what `CPU.cpp`'s `poke8_2068`/`poke16_2068` gate on.
- **DOCK/ROM image provenance.** Real `.DCK` dumps and ROM images may be copyrighted. Fine to use privately for bring-up/testing; be deliberate about what actually gets committed to a repo, especially if it's ever made public. Slice 4 operationalized this: policy is written down in `test/dock/README.md`, and `test/dock/.gitignore` excludes a local-only `real/` directory so local testing doesn't risk an accidental commit. The "ROM loading" increment put this into practice for real: a genuine TS2068 HOME ROM/EXROM (from a local reference library, not this repo) got read and tested against, `git status` was checked before every commit throughout this project specifically to catch this, and no ROM byte ever entered the working tree.
- **GPL-3.0 inheritance.** Any dependency or tool pulled in during this port needs to stay license-compatible with the GPL-3.0 base.
